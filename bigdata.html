Big data basic

Characteristics of big data

  • Volumn: sheer size of data.
  • Variety: complexity. structural (Ex. formats), media (Ex. how data is delivered), semantic (Ex. units), availability (Ex. real-time)
  • Velocity: speed at creating, storing, and analyzing data.
  • Veracity: quality.
  • Valence: connectedness.
  • Value: benefits to business.

Why DBMS rather than file system

  • Declaritive query language.
  • Data independence. (applications not worrying about data formats and locations)
  • Efficient access through optimization.
  • Data integrity and security.
  • Concurrent access.

BDMS

  • Flexible, semi-structure data model.
  • Support for common big data types.
  • A full query language.
  • Efficient parallel query runtime.
  • Wide range of query sizes.
  • Continuous data ingestion.
  • ACID is hard to maintain in BDMS, so we aim BASE (Basic Availability, Soft State, Eventual Consistency)
  • CAP theorem: distributed system cannot simultaneously achieve (consistency, availability, partition tolerance)
  • May not guarantee consistency. (most likely supports eventual consistency)

Semijoin (used in distributed settings)

  • A semijoin from R to S reduce data transmission cost.
  • 1. "Project" R on attribute A (call it R[A]).
  • 2. "Step" this projection from the site of R to the site of S.
  • 3. "Reduce" S by eliminating tuples where attribute A are not matching any value in R[A].

Big data system

Vector space model

  • Ex. document vector using inverse document frequency.
  • Similarity function (cosine distance)
  • Query term weighting.

Graph data model

  • Find the shortest path between two nodes.
  • Find optimal round-trip path that must include some specific nodes.
  • Find best "compromisable" path while cannot meet all the requirements simultaneously. (pareto optimality)
  • Communities: clusters within the graph with lots of edges connecting nodes within.

Data stream

  • Sequence of data records.
  • Small window of data elements.
  • Fast and simple computations.
  • No interation with data source.
  • Independent computations.
  • Near real-time.
  • Static: size determins time and space.
  • Streaming: unbounded size, but finite time and space.
  • Challenges: size and frequency of streaming data can be unpredictable, sporadic.

Lambda architecture

  • Real-time data is pushed to batch.

Data lake

  • Stores raw data.
  • Add data model on read. (schema on read)

Data warehouse

  • Well-defined structure.
  • Schema on write.

Redis

  • In-memory data structure store. (string, hash, list, set, sorted set)
  • ziplist: compresses list

Aerospike

  • Distributed NoSQL DB and key-value store.
  • Data types: scalar, list, map, geospatial, large object.
  • Ensures ACID.

AsterixDB

  • Designed for semi-structured data.
  • Supports multiple query languages.

Solr

  • Large scale text data search.

Vertica

  • Relational DBMS.

MongoDB queries

# SELECT * FROM Beers
db.Beers.find()

# SELECT beer, price FROM Sells
db.Sells.find({}, {beer:1, price:1})

# SELECT manf FROM Beers WHERE name = "Heineken"
db.Beers.find({name: "Heineken"}, {manf:1, _id:0})

# SELECT DISTINCT beer, price FROM Sells WHERE price > 15
db.Sells.distinct({price:{\$gt>15}}, {beer:1, price:1, _id:0})

# Count the number of manufacturers whose names have the partial string "am" in it. (case insensitive)
db.Beers.find(name: {\$regex:/^am/i}).count()

# Count the number of manufacturers whose names starts with "am" and ends with "corp".
db.Beers.find(name: {\$regex:/^am.*corp\$/}).count()

# Find items which are tagged as "popular" or "organic".
db.inventory.find({tags: {\$in: ["popular", "organic"]}})

# Find items which are not tagged as "popular" or "organic".
db.inventory.find({tags: {\$nin: ["popular", "organic"]}})

# Find 2nd and 3rd element of tags.
db.inventory.find({}, {tags: {\$slice: [1,2]}})

# Find a document whose 2nd element in tags is "summer".
db.inventory.find(tags.1: "summer")

# Count the number of unique addresses of drinkers.
db.Drinkers.count(addr: {\$exist: true})

# Get the distinct values of array.
db.countryDB.distinct(places)

# How many tweets have location not null?
db.users.find({"user.Location": null}).count()

# How many people have more followers than friends?
db.users.find({ \$where : "this.user.FollowersCount > this.user.FriendsCount"}).count()

# Perform a query that returns the text of tweets which have the string "http://".
db.users.find({"tweet_text" : {\$regex : ".*http://.*"}})

# Get all the tweets from the location "Ireland" which also contain the string "UEFA".
db.users.find({"tweet_text" : {\$regex : "UEFA"}, "user.Location" : "Ireland"})

Hadoop basic

HDFS (Hadoop Distributed File System)

  • Scalability and reliability.
  • Spread files to differnt nodes for parallel access.
  • Partitioning and replication for fault tolerance. (node failure doesn't cause files to become unaccessible)
  • NameNode for metadata (1 node per cluster), DataNode for block storage (each node per cluster)

YARN (Yet Another Resource Negotiator)

  • Resource manager for Hadoop.
  • While node manager is in charge of each machine, resource manager decides who gets what.
  • Allows applications to run on the same Hadoop cluster.

MapReduce

  • Programming model for Hadoop ecosystem.
  • Relies on YARN.
  • Not good for frequently changing data.
  • Consists of mapping, shuffling, and reducing.
  • Delegates large tasks to lots of nodes and aggregates the result.

Weakness

  • Small datasets.
  • Advanced algorithms.
  • Infrastrutural replacement.
  • Task level parallelism.
  • Random data access.
