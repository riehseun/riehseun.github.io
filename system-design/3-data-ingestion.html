# Requirement and Problem

What data sources are there? NAS drive, API, DB, etc
How is data produced? NAS - file drop, API - push and pull, DB - change data capture
What is size of data ingested per day? 10GB per day
What is growth rate of data? Assume 10x in 2 years =? 100GB per day
What is peak load? If evenly distributed, 10GB/day = 100KB/s, Assuming 20x peak, 2MB/s 
Is the data generated in batch or real-time? Both
What is latency requirement for real-time data to become available for consumption? under one second
How is data used? build ML models
How long should data be retained? 5 years
What are formats of data? csv, text file, parquet, JSON, XML, etc
In which format should the system store data? parquet
For real-time data, can Kafka be used? Yes
For processing batch data, can Spark be used? Yes
Should the system guarantee exactly-once or at-least-once delivery? At-least-once is needed

# Detailed Design

## Ingestion layer
Real-time
  Kafka
    Commit log, high throughput, decoupling ingestion from production
  Kafka topics
    Parition by high cardinality key (Ex. user_id) or hash of primary key 
    Partition count - start with 24-48
Batch
  NAS upload, periodic DB exports, Bulk API dumps
  Upload files into object storage
  Push metadata to Kafka
  Trigger batch processing

## Processing layer
Spark (structured streaming)
  Unified API for both batch and streaming
  Micro-batch model is suitable for less than 1 sec SLA
Responsibilities
  Schema validation
  Data cleaning
  Data de-dep
  Enrich data
  Write to storage
Event Id
  Every record must have globally unique event id and timestamp
De-dup
  Retention with TTL based on event_id
  Window should not be too large to save memory
Exactly-once
  Kafka offset tracking
  Checkpointing
  Atomic writes to storage
  Idempotent sink operation
Batch processing
  Backfills
  Heavy joins
  Feature engineering

## Raw data
Original format
Partition by ingestion date
Append only (data is never mutated!)

## Curated data
Parquet format
Partition by event_date and event_time

## Real-time serving (if ML models require dynamic features)
Option 1. query from streaming processor memory
Option 2. push to low-latency store (Ex. Redis)

# Data invalidation
