Parking Lot Reservation


# Requirement

User request parking reservation with specific time slots (start, end)
System assigns available spot automatically
Parking spot is released at the end of reservation
Parking spot could be under maintenance

20 users but expected to grow
5 parking spaces but expected to grow
Each user make 1 reservation request per month on average
Prevent double booking

# High-level

Request -> Service (->DB) -> Response

Write path
- Service checks if there is free slot in DB between start and end time
  - For all parking spots, query reservation table to see if there is active booking between start and end time (if result is null, then spot is free)
  - SELECT * FROM reservation where parking_id = ? AND start_time <= requested_end_time AND end_time >= requested_start_time
  - If above query returns result, that parking spot in unavailable
- If yes, create reservation and respond with slot #. If no, respond with reservation failure

Service
- A single virtual machine

DB
- A single SQL instance

# Discussion

Performance
- Index query that finds all available parking spots given start and end time
- parking id column must be indexed

Scalability
- Triggers:
  - Server: >70% cpu, >80% memory, latency > SLA, >10k number of connections
  - DB: >5ms read, >10k write per second
- Service: horizontally add nodes
- DB: partition/shard data where shard id is parking_id
  - Use consistent hashing to distribute partitions/shards to different data nodes

Availability
- GLB to route traffic to nearest edge
- Each edge has set of services and DBs
- Active-Active setup where GLB does healthchecks on edge and send traffic to live edge

Reliability
- Service fails? route to other service node
- DB fails? fallback to replicas (create replicas for each shard and spread them across different geo region)

Monitoring
- Metrics: latency, QPS, cache hit rate, cpu/memory utilization
- Tools: Prometheus for metrics, Grafana for dashborad
- Alerts: when metrics hit threshold

Consistency
- If using a single SQL instance, use strong consistency by leveraging ACID property of SQL
- If DB is distributed, choose consistency over availability
  - Do not commit transaction until R + W > N 

Synchronization
- If multiple users race against the same time slot
- If using single SQL, leaverage ACID property 
- If distributed, write goes in one shard in region A and read takes place in all other shards in region B,C,D,E
- If write shard dies, one of remaining shards become the write shard
- Write shard propagates updates to all other shards synchronously (use quorum to achieve strong consistency)
