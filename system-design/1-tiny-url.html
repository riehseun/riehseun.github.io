Tiny URL


# Requirement

Create short url from long url
Redirect short url to long url
Update or delete existing mapping

Scale from few users to billions
10:1 read-write ratio
End-to-end Latency - 300ms
High availability

# High level

Request -> Service (->DB) -> Response

Write path
- Service checks if short version already exists in DB
- If yes, return retrieved short url
- Else, compute short version, store it in DB, and return it

Read path
- Service read original url using short url from DB
- User is redirected to original url

Service
- A single virtual machine
- Assume there is 62 chars to use to generate URLs
- Number of short URLs = 62**n where n is number of chars
- Assume 20 users grow to 1B, write QPS grows to 10 per day, and data retension is 5 years
- Then, we need to have 62**n > 1B * 10 * 365 * 5
- Generate a sequence nubmer, then use base 62 conversion to generate random string
  - How to generate sequence number
  - Snowflake: timestamp + machine ID + sequence, distributed

DB
- A single SQL server

# API and Data

- GET /{short_url}
- POST /urls
{
  "url": original_url
}
- PUT /urls/{short_url}
{
  "url": original_url
}
- DELETE /urls/{short_url}

Table - URL
- url_id (bigint, pk)
- short_url (varchar, unique)
- original_url (text)
- user_id (bigint)
- created_at (timestamp)
- updated_at (timestamp)

Index
- short_url: optimize read path, duplication not expected so performance is good
- original_url: optimize write path, little duplication expected so performane is good
- Write is little slower having to update sorted data in memory

# Discussion

Performance
- Use cache in front of DB to handle read traffic fast
- Store frequently access data in key-value store
  - key: short_url
  - value: url
- Invalidate with TTL 
- Cache aside (if cache miss, read from DB and put it in cache, then return it)
- Eviction: least recently used

Latency
- Cache hit: 10ms
- Cache miss and DB query: 100ms
- Network on request/response: 200ms
- Overall <300ms

Scalability
- Triggers
  - Server: >70% CPU, 80% memory, latency > SLA, number of connections > 10K/s
  - DB: read >5ms, throughput >10K write per second
  - Cache: >20% cache miss, read > 1ms
- Server
  - Add nodes (horizontal scaling)
- Cache/DB
  - Shard/partition data using shard key
  - Shard key is url_id
  - Consistent hashing to distribute shards across different nodes

Availability
- GLB to route incoming requests to nearest edge
- Servers, DB, Cache are deployed to each edge
- Active-Active where GLB healtchecks periodically on edge servers, and route traffic to live edges

Reliability
- Service fails? route to other service node
- Cache fails? fallback to DB
- DB fails? fallback to replicas (create replicas for each shard and spread them across different geo region)

Monitoring
- Metrics: latency, QPS, cache hit rate, cpu/memory utilization
- Tools: Prometheus for metrics, Grafana for dashborad
- Alerts: when metrics hit threshold

Consistency
- If using a single SQL instance, use strong consistency by leveraging ACID property of SQL
- If DB is distributed, choose availability over consistency
  - Being redirected to a stale URL in rare cases is okay

Synchronization
- If multiple requests to create/update the same URL
- If single SQL DB, rely on built-in ACID
- If distributed DB, use only one region for write relying on ACID and rest for read (slow write, SPOF)
