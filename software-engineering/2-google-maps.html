Requirement

Functional

  • How many total users in the system? 1B
  • How many daily users in the system? 100M
  • What is the main function of the system? Location update, navigation, ETA, map rendering
  • What is the size of road data? terrabytes
  • Should the system consider traffic conditions? Yes
  • Should the system support different travel mode like driving, walking, public transit? Yes
  • Should the system support multi-stop directions? No
  • Should the system support business photos and videos? No
  • Should the system show the route as user moves? Yes
  • Should the system show new route when user deviates from the path? Yes

Estimation

Storage

  • Map tiling
    • Geo coding
      • Convert each address to a pair of latitude and longitude
    • Geo hashing
      • Encode geographical area with digits
      • Each geographical area is considered as a tile
  • Map rendering
    • Clients only download relevant tiles for the area
  • Metadata - negligible
  • Road info - assume terrabytes
  • World map
    • At zoom level 21, there are 2^21 = 4.3 trillion tiles
    • Assume each 256 by 265 tile image is 100KB
    • Then, higest zoom level would have 430PB
    • Assume 90% of world surfaces are ocean, desert, mountains, etc
    • Then, 430PB * 0.1 = 43PB
    • The subsequent zoom level would have \( 43 + \dfrac{43}{4} + \dfrac{43}{16} + \dfrac{43}{64} + \dots \) = ? PB

Bandwidth

  • Assume
    • Assume each user uses navigation 1 hour per day
    • Then, 100M * 1hr = 100M hours per day = 360B seconds per day
    • Assume each GPS update is sent every 10 seconds on average
    • Then, 360B / 10 = 36B GPS updates per day
    • Assume both request and response is 100 bytes
  • Incoming/Outgoing - 36B * 100 bytes * 8 / 86400 = ? bps

High level design

Location finder

  • Finds current location of user and show it on map

Route finder

  • Finds path between two points
  • Shows the path on the map for the user

Navigator

  • Tracks user journey
  • Sends updated directions when user deviates from the path

Distributed search

  • Finds lattitude and longitude from the place name

Area search service

  • Receives lattitude and longitude of source and destination from distributed search
  • Finds area spanning these two points

Graph processing service

  • Runs shortest path algorithm

Graph DB

  • Holds road data

Kafka

  • Listens to events and triggers services accordingly
    • For example, when user deviates from the path, it calls area search service to compute new optimal path
  • Collections streams of location data for different users
    • This data is used to understand traffic patterns

Workflow

  • User enters start and end point
  • Current location of user is found by location finder and this can be the start point
  • User request is forwarded to route finder service
  • Route finder forwards requests to area search service
  • Area search service finds the latitude/longitude of start and end location from distributed search
  • Area search service computes the area spanning the two points
  • Graph processing service finds the shortest path and return it to route finder
  • Route finder displays the optimal path to user
  • Nagivator updates user location on the map as user is moving. It also provides directions to user
  • When user deviates from the path, Kafka alerts area search service to recalculate the optimal path

API

  • curr_location(location)
    • Display current location of user on the map
    • location
      • Boolean to indicate if user location is turned on or off
      • If off, the service will ask user to turn on the location
  • find_route(start, end, transport_type)
    • Computes optimal route between two points
    • start
      • Start point in text format
    • end
      • Start point in text format
  • directions(curr_location, steps)
    • Provides alerts of when and where to make turns

    • segment_id (int)

Relational DB

Detailed design

Add segment

  • Each segment has its lat/long boundary coordinates and the graph of road network
  • Segment adder assigns unique ID to each segment
  • Segment adder forwards segment information to server allocator
  • Server allocator maps segment to server, hosts the graph on the server, and returns server ID
  • Segment adder stores segment to server mapping in key-value store. It also stores segment's lat/long boundary coordinates

Handle user request

  • User provides source and destination
  • Distributed search finds lat/long coordicates of source and destination
  • Graph processing service finds the segment from lat/long coordicates
  • Graph processing service finds the server hosting the graph using segment ID
  • Graph processing service runs shortest path algorithm

Improve estimations using live data

  • Websocker provides persistent connection between user devices and the servers
  • Load balancer balances the connection load
  • Kafka collects location data from all servers
  • Data analytics engine uses ML to improve ETA. Then, it publishes result to pub-sub topic
  • Map update service listens for updates and updates segment graph
  • Graph preprocessing service recalculates the new paths on the updated segment graph

Sub-graphs (Segment)

  • Computing distances between two points on the entire graph is super expensive
  • Precalculate the shortest distance between two points in sub-graphs. Also, cache most requested routes
  • Connect sub-graphs and run the shortest distance algorithm

Segment adder

  • Segment adder assigns unique ID to each segment
  • Segment is assigned to a server and segment adder gets the server_id
  • Key-value store keeps the segment to server mapping

Evaluation

Availability

  • Replication
    • Replicate each server and DB
  • Fail over
    • Active sends heartbeat to passive periodically
    • When heartbeat stops, passive takes over active's IP

Reliability

  • Data loss
    • Replicate entire components to different zone or geographical region

Scalability

  • Horizontal scaling
  • Data partitioning

Performance

  • Use of sub-graphs make computation faster than using entire graph
  • Cache processed sub-graphs
