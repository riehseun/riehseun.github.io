Estimation

Server

  • Assume
    • 10B queries per day = 115740 queries per second
    • A single server can handle 10K connections
  • 115740 / 10k = 12 servers are needed

Storage

  • Assume
    • There are 10B queries per day.
    • There are 1B unique queries out of 10B per day.
    • Each query has 10 characters on average.
    • Each character requires 2 bytes to store.
  • 1B per day * 10 * 2 = 20GB per day.

Bandwidth

  • Assume
    • Suggestions have the same length, which is 10 chracters.
    • Top 10 suggestions are shown to users.
  • Incoming: 10B * 10 * 2 * 8 / 86400 = 18.5Mbps
  • Outgoing: 10 * 10 * 18.5Mbps = 1.85Gbps

High level design

  • Add seed URLs to URL frontier
  • Service host fetches URLs from URL frontier
  • Service host gets IP addresses of URLs from DNS resolver and starts downloading
  • Content parser validates HTML pages
  • Duplicate HTML eliminator checks if HTML is already in storage
  • Duplicate URL eliminator checks if URL is already in storage
    • If URL is not duplicate, it is added to URL frontier

Seed URLs

  • Base URLs

URL frontier (priority queue)

  • Contains all the remaining URLs to download

Service host

  • Each worker dequeues URL from the URL frontier
  • Each worker uses DNS resolver to acquire web page's IP address

DNS resolver

  • Custom DNS resolver is needed because DNS lookup is time consuming process
  • Cache frequently used IP addresses within their TTL

Content parser

  • HTML is parsed and validated

Duplicate HTML eliminator

  • Checks if document is duplicate by checking the hash values of two web pages

URL extractor

  • Extracts links from HTML pages

URL filter

  • Exclude certain content types, file extensions, and blacklisted sites

Duplicate URL eliminator

  • Bloom filter or hash table is used

URL storage

  • Stores already visited URLs

Detailed design

URL frontier (priority queue)

  • Use BFS (DFS is not a good choice because depth can be very deep)

Priority

  • Prioritizer computes priority of each URL
  • Each queue is assigned a priority
  • Queue selector has higher probability of picking queues with higher priority

Politeness

  • Download one page at a time from the same host
  • Each thread in service host has a queue and download URLs from that queue
  • Queue router ensures that each queue only contains URLs from the same host
  • Mapping table maps host (ex. wikipedia.com) to queue
  • Queue selector rotates worker threads that downloads webpages

Freshness

  • Recrawl is needed because webpages are constantly updated
  • Recrawl URLs with higher priority first

Storage

  • Due to huge size of URLs, need to store URLs into a disk
  • But, disk is slow, thus introduce buffers in memory
    • Enqueue buffer - once filled, written to disk
    • Dequeue buffer - keeps cache of URLs to be visited. It periodically reads from disk to fill the buffer

Service host

Robots.txt

  • Specifies pages that are not allowed to be downloaded

Performance optimization

  • Crawl jobs are distributed into multiple servers, and each server runs multiple threads
  • Response from DNS resolver is cached in Service host
  • Place crawl servers close to website hosts in terms of geographical region
  • Enforce maximum wait time from the host

Robustness

  • Use consistent hashing to distribute load among crawl jobs
  • Save crawl states into disk (When failure happens, can resume from the saved state)

Typeahead

High level design

  • Top 10 suggesstions are queried from Redis cache.
  • User searches are stored in NoSQL DB.

API

  • get_suggestions(prefix)
  • add_to_database(query)

Component design

Suggestion service

  • Returns top 10 from cache. (Redis)

Collection service

  • Stores metadata about user query.

Aggregator

  • Retrieves data from HDFS and distributes to different workers.
  • MapReduce aggregates frequency of prefixes in given interval and frequency is updated in Cassandra.

Trie builder

  • Creates/updates tries.
  • Stores tries in DB via ZooKeeper.

Trie structure

  • Store this data structure in memory. (Not in DB)
  • Assume case insensitive.
  • For example, when user types "CAP", node "P" will show all suggestions.

How to find top suggestions

  • Store the number of searches in the last phrase.
  • Parent nodes recursively call child nodes to get suggestions and counts.

How to update trie

  • Spread copies of tries on each server and update them offline. (While some nodes are serving the traffic)
  • Also, remove inappropriate terms from the tries duing this time.
  • Also, update frequency at each node.

Client behavior

  • Client should hit the server only when no key has been pressed for 50ms.
  • When user is typing, client should cancel in-progress request.
  • Client should wait until at least few characters are entered in the beginning.
  • Client can store recent history of top suggestions.
  • Establish connection between client and server as soon as browser opens up.
  • Server should push some part of cache to CDNs.

Data partitioning

  • Range based partitioning
    • Pick storage based on starting character of queries.
    • Results in uneven servers.
    • There will be hot servers.
    • May not be able to fit the size of data into each server.
  • Hash based partitioning
    • Each term gets a server number from hash function.
    • Minimize hot servers.
    • Must query all servers and aggregate the result.
