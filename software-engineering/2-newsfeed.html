Estimation

Server

  • Assume 100M daily active users
  • Assume a single server can handle 10K connections
  • 100M / 10K = 10,000 servers are needed

Storage

  • Assume daily acive users is 100M and total users is 1B
  • Assume each user metadata is 100KB
  • Assume each user has 1000 posts
  • Assume each post 1MB (text, image, video combined) on average
  • 100KB * 1B = 100TB for all user metadata
  • 1000 * 100M * 1MB = 100PB for all posts

Bandwidth

  • Assume each user creates 1 posts per day
  • Assume each user views feeds 10 times a day
  • Incoming - 100M * 1 per day * 1MB = 100TB * 8 / 86400 = 9.26Gbps
  • Outgoing - 100M * 10 per day * 1MB = 1PB * 8 / 86400 = 92.6Gbps

Storage schema

Chat history

  • RDBMS (MySQL) or NoSQL (MongoDB)
    • Not ideal because we need to do frequent read/write
  • Key-value store
    • Horizontal scaling is easy
    • Low latency when accessing data
    • Wide-column DB like HBase
      • HBase stores data into memory buffer and move data into disk when buffer is full, which works well with variable-sized data

Message

  • message_id - int
  • message_from - int
  • message_to - int
  • content - string
  • created_at - timestamp

Group message

  • channel_id - int
  • message_id - int
    • Use local sequence number generator because IDs only need to be unique within a group
  • used_id - int
  • content - string
  • created_at - timestamp

Newsfeed

High level design

  • Feed is generated via aggregating friends and followers posts.
  • Feed is published and stored in cache and DB.

API

  • generate_feed(user_id)
  • get_feed(user_id, count)

DB

  • Use relational database.
  • Use a graph database only for the relationships between users.

Entity

  • entity_id (int, pk)
  • name (varchar)
  • name (description)
  • creation_date (datetime)

User

  • user_id (int, pk)
  • name (varchar)
  • email (varchar)
  • creation_date (datetime)

  • feed_item_id (int, pk)
  • user_id (int)
  • entity_id (int)

Relationship (Edge)

  • relation_id (int, pk)

Component design



Newsfeed generation service

  • Generate feed offline: dedicated servers that generate user feed and store them in memory.
  • Retrive ID of all users and entities that the user follows.
  • Retrive the latest, most popular, most relevant posts for those IDs.
  • Ranking service ranks these posts.
  • Newsfeed cache stores feed into cache.

Newsfeed publishing service

  • Fetches a list of post IDs from the newsfeed cache, which has (post_id, user_id)
  • Complete data is pull from user and post cache to construct feeds.

Notification service

  • Pull: users pull feed at regular interval.
  • Push: post is pushed to all followers whenever it is published.
  • Celebrity users should have pull model.
