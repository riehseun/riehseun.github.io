# You faced significant technical obstacles

I was machine learning engineer at Layer6 AI, AI specialized org within TD Bank
My team was responsible for ML serving design, MLOps, and ML infrastucture
My team was using Azure Databricks for computing inference and ground truth

In Dec 2023, all ML use cases were using Databricks interactive clusters
Interactive clusters will shut down after certain inactivity, during which time the cost will incur
I wanted to use job clusters because they would terminate immediately after job finishes

Because we used Azure Data Factory to trigger Jobs in Databricks, natural solution was to configure Job Clusters on Azure Data Factory UI
However, TD internal process stated that any change to Data Factory must be made by TD internal pipeline and creating chagne order
Doing this for each use case would take significant time and effort
I designed a workaround such that instead of configuring job clusters on Data Factory UI, I store configuration files in DBFS
When job is run, it would first read onfiguration file in DBFS, and create and run jobs based on that configuration using Databricks API
Instead of deploying Data Factory changes for all use case, I could just deploy one notebook in ADB which creates the configuration files in DBFS

The workaround was accepted by the team and team successfully started using job clusters
47k - 70k cloud cost savings per year for each use case in production (9 use cases in total in production)


# You faced significant technical obstacles

I was machine learning engineer at Layer6 AI, AI specialized org within TD Bank
My team was responsible for ML serving design, MLOps, and ML infrastucture
In mid 2022, My team used Elasticsearch for storing inference and ground truth data, that is used by our internal application

The cluster was deployed in 3 virtual machines
Elasticsearch was outdated at version 6
It was free version and did not support authentication
Write was slow, taking more than 3 hours during inference and ground truth
Disk was nearly full

In Dec 2022, upgraded Elasticsearch version from 6 to 8
Capacity estimate for the next 5 years
Deployed the cluster in kubernetes to leverage auto-scaling and soother integration with inference/gt job also running in Kubernetes
Higher compute and lower storage to hot nodes and lower compute and higher storage for cold nodes
Splited each Elasticsearch index into many shards to achieve parallel reads (Each shard did not exceed 600MB)
Replicated each index into different data nodes to achieve availability
Took a periodic snapshot of data and store them into storage account to achieve reliability
Taking snapshot of data from version 6 and restoring it in version 8 word in DEV but not in prod (due to size)
Thus, I had to do research in the midst of production deployment and leaverage a method called reindexing in Elasticsearch

2x in read/write
Learned to leave lots of time for production deployment for large scale data application


# You faced significant technical obstacles

I was working for cloud engineering team, developing Enterprise pipeline for code and infrastructure
Users were raising concerns about usuability of the pipeline
In October 2021 at TD, when mistake is made in pipeline configurations, users had to go back to the beginning

No way to update deployment configuration in the middle
Could not update configurations in the tool because audit said source of truth had to come from git
Deployments were taking over one week on average because of this

New pull request containing release ID of the ongoing deployment
New pull request containing updated configuration
When the new pull request is merged, the solution identified the original release and updated it configuration

Users reported deployment time got reduced by twice and now take 3 days on average
