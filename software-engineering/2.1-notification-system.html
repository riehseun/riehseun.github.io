High level design

Components

  • Services
    • Triggeres notification sending events
    • Can be micro-service, cron job, etc
  • Notification servers
    • Validates email, phone number, etc
    • Queries cache or DB to fetch data to render a notification
  • Cache
    • Caches user info, device info, notification templates
  • Message queues
    • Remove dependency between notification servers and third party services
    • Serves as buffer when high volumn of notifications are sent out
  • Workers
    • Pull notification events from message queues and send them to corresponding third party services
  • APNs
    • Apple push notification service
  • FCM
    • Fire-base cloud messaging
    • Used for Android device
  • SMS service
    • Ex. Twilio, Nexmo
  • Email service
    • Ex. Sendgrid, Mailchimp

Workflow

  • Service calls APIs to notification servers
  • Notification servers fetch metadata from cache or DB
  • A notification event is sent to corresponding queue
  • Workers pull nofitication events from message queues
  • Workers send notifications to third party services
  • Third party services send notifications to user devices

Detailed design

Workers

  • Retry
    • Workers store notification data in notification log DB
    • When third party fails to send notification, the notification is added to message queue for retrying

Notification servers

  • De-duplication
    • Distributed nature of the design cannot guarantee exactly-once delivery
    • When notification arrives, check event ID. If event ID is seen before, discard it
  • Performs authentication
  • Serves as rate limiter

Analytics service

Evaluation

Functional

  • Business objective
    • Design supports push notification, SMS, email via different third party service providers
    • Notification table stores whether user opt-out from notification or not

Non-functional

  • Availability
    • Replication
      • Replicate each
      • Replicate entire system to different zone or geographical region
    • Fail over
      • Option 1. active-active
        • Both actives manage traffic, spreading load between them
      • Option 2. active-passive
        • Active web servers, cache send heartbearts to passives
        • If heartbeats stop, passives take over actives' IP and resume service
  • Reliability
    • Data loss - periodically take snapshot of data and store snapshots in multi zone or region to prevent data loss
    • Handling failure - implement retry mechanism
  • Scalability
    • Horizontal scaling
    • Data partitioning
  • Performance
    • Peak load
      • Scale notification servers to handle increased requests from clients
      • Scale workers to deque messages fast from message queues
    • Message queues allow async processing where producers do not need to wait for consumers to finish
  • Consistency
    • If device info is out of sync, then message is likely sent to device that no longer exists
    • If notification info is out of sync, then message could get delivered to users who opted-out
    • Both cases are not detrimental (although it is degrading) thus eventual consistency should suffice
