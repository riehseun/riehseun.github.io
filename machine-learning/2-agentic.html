# Design Agentic RAG system

Plan - clasifies query using LLM and generates execution graph
  Simple question -> direct LLM
  Knowledge question -> RAG
  Transactional -> API/DB call
  Complex question -> Decompose input into multi tasks -> Execute each task (RAG) -> Verify -> Terminate (Ex. loop detected, max number of call reached, max cost per query reached, confidence threshold reached)
Execute - run tools
  API calls, DB queries
  Retrieval - select between cheap and strong model based on confidence level of retrieved document, when conflicting documents are retrieved, select best document based on freshness and authoritativeness of documents
  Evaluation - when RAG output degrades measure by offline evaluation, find chunks from poorly generated reponse and mark that chuck as stale
Verify
  Self-critic final generated response
Terminate
  Terminate if cycle is detected in the execution graph or when target confidence is met

Ingestion
Parse - converting input document into Markdown to minimize tokens
Chunk - semantic chunking and generate embedding for each chunk
Vector DB - mapping between embedding and chunk

RAG
Query understanding - correct spelling mistakes, summarize prompt to reduce token, and generate input embedding
Retrieval - search vector DB and find top k chunks that has closest embedding distance to input embedding (ANN - HNSW)
Re-ranking - rank retrieved chunks based on relevance to input embedding
Output - combine system prompt, chunks, and input query and generate response

Cache
Query understading - use embedding cache for input
Retrieval - use cache to store retrived document for input query
Output - use cache to store generated response for each combo of input query and chunks

Evaluation
Retrieval - recall at k
Re-ranking - NDCG
Generation - Answer faithfulness, groundedness, context precision/recall
End-to-end - task success rate

Discussion

Confidence of retrieved document is determined by cosine similarity between input query and chunk embedding, normalized per query
If distribution between top-1 and top-k is flat, we have low-confidence
If retrieved docuemnts have conflicting information, we have low-confidence
Alternatively, use light weight model to check if chunks have enough information to answer user query

To deal with low-confidence senario
Increase k, use different retrieval strategy, re-run ANN 
Then, we use stronger model and tell the model to say don't know if it is not very confident about the generated output 

# Design Agentic fraud investigation system

ML serving
  Rule based detection - Simple logic to automatically decline transaction (Ex. blacklisted users/card) 
  Logistic regression to decline transaction that has very high confidence >98% 
  XGBoost algorithm to pass/review/decline transactions because on confidence level

Agent kicks in upon detecting potential fraud

Plan
  Agent detemines relevant features (past transaction history, device, geo locations)
Execute
  Agent gathers relevant features via API calls and DB queries
  Agent generates explanation for determining fraud vs non-fraud
Verify
  Agent self-critique its conclusion of investigation
Terminate
  Agent terminates loop when multi-cycle is detected, token budget for each investigation runs out, confidence level for a conclusion reaches a threshold
