# Design an AI agent system for TD Bank that automates fraud investigation.

Typical model serving pipeline 

Transaction -> Rule based detection -> Light model -> Heavy model -> Decision -> Re-training

Rule based detection 
  Agent runs a software with simple logic to automatically decline transaction (Ex. blacklisted users/card) 
Light model 
  Agent runs a logistic regression to decline transaction that has very high confidence >98% 
Heavy model 
  Agent runs XGBoost algorithm to pass/review/decline transactions because on confidence level 
Decision 
  Plan - agent detemines relevant features (past transaction history, device, geo locations)
  Execute - agent gathers relevant features via API and DB queries
  Reason - generate response explaning the decision
Re-training
  Agent triggers re-training when new batch of data become available

# How do you detect and prevent incorrect reasoning and halluciation in production?

Prepare careful system prompt to train LLM to say it does not know when it is not absolutely confident about the response it generates
Provide steps by step instructions to LLM rather than giving multiple tasks in one prompt
Also, ask LLM to perform self-verification before generating final response

Alternatively, use RAG and say "only use retrieved document to generate response, and if no relevet doc is found, answer do not know"

Use newest version of LLM although there is trade-off with cost

# Build Agentic RAG system

Document ingestion -> Parsing and chunking -> Vector DB
Query -> Query expansion -> Retrieval -> Re-ranking -> Output

Parsing and chunking
  Agent convert documents to markdown format to minimize total number of tokens to minimize cost
  Chunking - agent split documents by sentences and use SentenceBERT to compute embedding
Vector DB  
  Store mapping between embedding and chunk
Query expansion
  Agent correct spelling mistakes, and additional context to input prompt
  Agent compute embedding of input prompt
Retrieval
  Agent searches Vector DB to retrieve top k relevant chunks using ANN (Ex. HNSW)
  If conflict documents are retrieved, agent calls LLM to remediate them
Re-ranking 
  Agent re-ranks chunking based on relevance to query, freshness and trust level of document
Output
  System prompt + input query + most relevant chunks are sent to LLM to genereate response
  If LLM is not confident about outputs due to trust level of chunks, either say it does not know or agent resume from retrieval stage with different strategy

Retrieval
  Agent computes recall at k to measure percentage of actual relevant documents retrieved
Re-ranking
  Agent computes NDCG to measure the ranking quality of chunks 
Output
  Agent computes BLUE, ROGUE to compute precision and recall against reference outputs
  Agent also computes factual accuracy

Self-improve
  When RAG is degrading (by observing evaluation metrics), agent identifies chunks retrieved for generating poor response
  Agent identifies chunks in vector DB and mark them as stale

# Optimize Agent RAG serving 1M queries per day

Embedding cache - during query understading, read from cache rather than computing embedding of input query
Retrieval cache - for known input query, use cached top k chunks
Output cache - cache with TTL for known input query and chunk combo
In retrieval step, if confidence is high, use cheaper model, if confidence is low, use stronger model
Squeeze tokens - use cheap model to summarize inputs, de-dup chunks, limit final chunks to less than 3
