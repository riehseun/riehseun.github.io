# Build Agentic AI system for RAG

Document
Parse - converting input document into Markdown to minimize tokens
Chunk - typically, split document by sentence, and use SentenceBERT to generate embedding
Vector DB - mapping between embedding and chunk

Input query
Query understanding - correct spelling mistakes, summarize prompt to reduce token, and generate input embedding
Retrieval - search vector DB and find top k chunks that has closest embedding distance to input embedding (ANN - HNSW)
Re-ranking - rank retrieved chucks based on relevance to input embedding
Output - combine system prompt, chunks, and input query and generate response

Cache
Query understading - use embedding cache for input
Retrieval - use cache to store retrived document for input query
Output - use cache to store generated response for each combo of input query and chunks

Evaluation
Retrieval - recall at k
Re-ranking - NDCG
Output - BLUE, ROGUE (precision, recall based on reference genereation), factual accuracy

Agent
Select between cheap and strong model based on confidence level of retrieved document
Self-critique generated response
When conflicting documents are retrieved, select best document based on freshness and authoritativeness of documents
When RAG output degrades measure by offline evaluation, find chunks from poorly generated reponse and mark that chuck as stale
Not only queries vector DB for retrieval, but also queries internal DB and APIs to retrieve relevant information if appicable

Agent workflow
Simple question -> direct LLM
Knowledge question -> RAG
Transactional -> API/DB call
Complex question -> Decompose input into multi tasks -> Execute each task -> Verify -> Terminate (Ex. loop detected, max number of call reached, max cost per query reached, confidence threshold reached)

Discussion

Confidence of retrieved document is determined by cosine similarity between input query and chunk embedding, normalized per query
If distribution between top-1 and top-k is flat, we have low-confidence
If retrieved docuemnts have conflicting information, we have low-confidence
Alternatively, use light weight model to check if chunks have enough information to answer user query

To deal with low-confidence senario
Increase k, use different retrieval strategy, re-run ANN 
Then, we use stronger model and tell the model to say don't know if it is not very confident about the generated output 

# Design an Agent system for fraud detection

Rule based detection 
  Agent runs a software with simple logic to automatically decline transaction (Ex. blacklisted users/card) 
Light model 
  Agent runs a logistic regression to decline transaction that has very high confidence >98% 
Heavy model 
  Agent runs XGBoost algorithm to pass/review/decline transactions because on confidence level 
Decision 
  Plan - agent detemines relevant features (past transaction history, device, geo locations)
  Execute - agent gathers relevant features via API and DB queries
  Reason - generate response explaning the decision
Re-training
  Agent triggers re-training when new batch of data become available
