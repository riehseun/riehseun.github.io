Fraud Detection


# Requirement

What kind of fraud should we detect? Fradulent financial transactions
Are there training data available? Yes, 10k fraud, and 1M non-fraud data available
Should model explain to users why transaction is declined? Yes
What is impact of false positive? Valid transaction decline, which drops user engagement
What is impact of false negative? Fraud goes through, damaging reputation of platform

How fast should inference be? Under 1s
How many transactions occur per day? 1M, 1M / (24 * 3600) QPS
Can peak QPS be 20-30x the average? Yes
Should model be continously trained? Yes

# Problem

ML - given a transcation, determine whether it is fraud or not, subject to minumum precision (we don't want to decline every transaction)
Business - minimize total number of fradulent transactions

# High-level

Transaction -> Rule-based detection -> ML-based detection with light model -> ML-based detection with heavy model -> Output (pass/decline/review) -> Feedback and re-train

Rule-based detection
  Use simple rule such as known fraud users and cards, or unreasonably large amount of transaction
ML-based detection
  Light model - decline if > 95% confident, otherwise pass to heave model for further review
  Heavy model - decline if > 90%, human review if between 90% and 70%, pass if < 70%

# Data 

We have training data, that is imbalanced. Undersample negative labels
Use result from human review for re-training

# Feature

Account
  Date of creation - convert to number by computing (inference date - creation date)
  Balance - compute average month-end balance for the last 3, 6, 12 months to make feature more robust against distribution drift
  Remaning credit amount
Transactions
  Number of transactions in last 3, 6, 12 months from user initiating current transaction
User
  City, country
Temporal
  Speed from last transaction

# Model

Light model
  Logistic regression to perform binary classification to determin whether transaction is fraud or not
Heavy model
  XGBoost - feature scaling not needed, very explainable, but relatively more expensive to re-train
  Feature selection
    Too many features become tech debt and increases inference time
    Run XGBoost model with each feature to compute predictive power
    Rank features by predictive power
    Add features one by one to the model (if adding more features does not improve the model, features are correlated, remove those features from feature set)
    Trim features down to 40-50
  Do cross-validation
    Split data into 20% test, 15% validation, 65% training set
    Test set must come (time-wise) after validation and training set
    Split training set into k bins - one bin for validation and all others for training set
    Population must not co-exist between validation and any of training set

# Discussion

Availability
  Replicate each service in different geo location (trade-off with cost)
Reliability
  Partition data into shards, and replicates shards into different geo location
Scalability
  Horizontally scale each service based on traffic pattern
Performance
  Model optimization - quantization (use fewer bits to represent number)
Monitoring
  Model - score drift, feature drift (PST), Kendall's Tau Rank Correlation (measures changes in rank of feature), ShapBand (measure tha deviation of SHAP values from baseline to inference data for each feature)
  System - latency, QPS

Offline feature store
  Store review result and new transaction/account/user record into offline feature store for re-training
  Ex. S3
Online feature store
  Fast reading of features during inference time
  Must have indentical feature transformation logic with offline feature store to avoid model degradation
  Ex. Redis

Explainability
  SHAP value

Real-time data ingestion
  Kafka for ingestion and Spark Structured Streaming for processing
  Transformed features are stored in both online and offline store for inference and re-training
  Features are aggregated in certain time window to ensure point-in-time correctness
