Harmful Content Detection


# Requirement

Where do harmful contents appear? Like posts in social media feeds?
What kind of harmfulness should the system detect? Nudity, violence, hate
Should the system support multiple languages? Yes
What is impact of false positive? Users lose engagement due to valid upload getting removed? Yes
What is impact of false negative? System gains bad reputation due to harmful posts going viral? Yes
What does a post consist of? Does it include text description, tags, images, videos, comments, etc? Yes
Can users report posts as harmful? Yes
Should model explain to users why posts got removed? Yes

Should harmful contents be removed immediately? For violence yes, for nudity and hate, system can take some time
How fast should inference be? Under 1s
How many posts does each user upload per day? 1 per day
How many daily active users are there? 100M, (1 * 100M) / (24 * 3600) QPS
Can peak QPS be 10-20 times the average? Yes
Should model be continously trained? Yes

# Problem

ML
  Input - a post
  Output - whether the post is harmful (violence, nudity, hate) or not
  Multi-class classification 

Business
  Minimize total number of harmful impressions, subject to certain precision (we don't want to remove all posts just to minimize impressions)

# Serving

Post -> Rule-based classification -> Light model -> Heavy model -> Review -> Monitoring model

Rule-based classification (~50ms)
  Simple rule such as known harmful keyword, image, video

Light model (~100ms)
  Reject post only if model has very high confidence
  Be strict on violence contents while be little lenient towards nudity and hate

Heavy model (~500ms)
  Reject if high confidence
  Send for review if medium confidence
  Pass if low confidence

Monitoring model
  Model to watch posts based on its new (dynamic) features like text, image, video added to comment section or updated in post itself

# Data

Postive - user reports, create synthetic harmful images (self-supervision), result from human review
Negative - result from human review, posts that have not been deemed harmful after it goes live in production

# Feature

Post
  Image, thumbnail image - CLIP's visual encoder
  Video - split into frames and use image encoder
  Text description, tags - distlimBERT (multi-language)
  Comments - sample them, then take weighted averge embedding, discounted by time

Author
  Previous violation count, total number of posts uploaded by author - normalize
  Country, city, language - embedding
  Account creation date - convert to integer representing (current date - creation date), then normalize

# Model

Light model
  XGBoost - struggles with embedding
  DNN with one model per harmful category - requires lots of data
  DNN with one model for all category 
    Gather all post and author features
    Training where total loss is weighted sum of losses from each category (violence, nudity, hate)
    Use focal loss to deal with class imbalance
    Use classification head during inference

Heave model
  Transformer-based encoder where input is combined feature set and output is handled by softmax for multi-classification

# Evaluation

Recall - percentage of actual harmful contents that model was able to detect
Precision - out of positive predictions, what percentage of them were correct 
PR curve - recall vs precision

Total number of harmful impressions
Total number of user reports
User churn rate

# Discussion

Availability
  Replicate each service in different geo region (trade-off with cost)
Reliability
  Parition data into shards, create multiple replicas of each shard, spread them across different geo region
Scalability
  Hortizontally scale each service based on traffic forecast
Performane
  Quantization - use fewer bits to represent number
  Pruning - in DNN, use drop-out to randomly shutdown neuron to make network more sparse
  Attention approximation - in transformer, avoid quadratic computation by approximating it with linear computation
  Multi-head attention - split computation into parallel running GPUs, and aggregate result
Monitoring
  System - latency, QPS
  Model - feature drift, score drift

Real-time data ingestion
  Kafka to handle real-time data and Spark Structure Processing to transform them
  Store features into both online and offline store, ensuring that transformation logic is identical
  Use a fixed-sized window to aggregate features (point-in-time correctness)
Offline feature store
  Used for training 
  Store data in columar format for fast feature level access (Ex. parquet)
  Use persistant storage (Ex. S3)
Online feature store
  Used for inference
  Store data in cache for fast read (Ex. Redis)

Continuous training
  A/B testing - direct traffic to candidate, and if candidate performs better than production, direct more traffics to candidate
