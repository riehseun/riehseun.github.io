Harmful Content Detection

  • Harmful Content Detection

# Requirement

Where do harmful posts potentially exist? Posts in Instagram
What kinds of posts are considered harmful? Nudity, violence, hate
Can harmful contents be in form of texts, video, images? Yes
How do harmful posts get created? Users upload them
Should the system consider multiple languages? Yes
What is impact of false positive? Users lose engagement because their uploads are deleted
What is impact of false negative? Users will view harmful contents, and platform will be flagged

How fast should the inference be? Under 500ms
How many users are there? 100M daily active users
How many posts does each user upload per day? 1, (1 * 100M) / (24 * 3600) QPS
Can peak QPS be 20-30x the average? Yes
Should model be continuously trained? Yes

# Problem

ML - given a post, determine whether post is harmful or not
Business - minimize total number of harmful impressions, subject to minimum precision (we don't want to flag every posts as harmful to minimize harmful impressions)

# High-level

Post -> Rule-based detection -> ML-based detection -> Decision (pass/delete/review) -> Intervene -> Feedback and re-training

Rule-based detection (50ms)
  Use known keywords, sentences, images, videos that represent harmfulness
ML based detection (400ms)
Decision (50ms)
  Pass/review/delete
  High precision is need for nudity
  High recall is needed to violence and hate
  For human review, prioritize violence, and also contents with higher confidence of harmfulness
Intervene
  New posts should be lower in the rank to minimize exposure in case model and human missed harmful content
  ML-based detection model also runs after the post goes live to inference based on real-time features

# Data

Positive - if review determines that post if harmful, if posts passes the model but later found to be harmful from user reports
Negative - if posts passes the model and not reported as harmful by users 
Use self-supervision to create synthetic texts/images/videos that are harmful
Use false negative post as trainging set
Use posts that are generated to fool the model

# Feature

Post
  image, video - CLIP
  text - distlimBERT
  comments - sample and compute average embedding, discounted by time
  number of reports - normalize
Author
  number of previously uploaded harmful posts - normalize
  account creation date - convert to integer representing (current date - creation date), then normalize

# Model 

Baseline
  Logistic regression - use features discussed and do binary classification to determine if posts is harmful or not

Use DNN for handling embeddings
  Separate models for each classification (nudity, violence, hate) - requires lots of training data
  One model for nudity, violence, hate but different classification head for each - pick this approach
  Fused features and feed them to a single model
  Combine embeddings from encoders (also combine numerical encoders) and use them in each classifier
Loss - cross-entropy (log-loss), create weighted loss of individual loess from nudity, violence, hate
Data imbalance - under-sample negative ground truth labels

# Evaluation

Recall at fixed precision - percentage of actual harmful posts that model was able to retrieve
Precision - percentage of predicted harmful posts that were correct
Accuracy - can't be used as data set is likely imbalanced
PR-curve

Total number of reports by users on harmful posts
Total number of impressions on harmful posts

# Discussion

Availability
  Replicate each service in different geo region (trade-off with cost)
Reliability
  Partition data into shards and create replicas for each shard, then store them in different geo location
Scalability
  Horizontally scale each service based on traffic patterns
Performance
  Model optimization - quantization (fewer bits to represent number)
Monitoring
  System - latency, QPS
  Model - feature drift, score drift

Offline feature store
  Store all processed features of author and posts (when they are uploaded) for re-training
  Ex. S3
Online feature store
  Store for fast-read during inference
  Processing logic must be idential with offline feature store to prevent model degrading
  Ex. Redis

Real-time data ingestion - use Kafka for ingestion and Spark Structured Streaming for processing, store processed features to both online (for inference) and offliun (for training) feature store
Continuous training - A/B testing to direct 10-15% of traffic to candidate, and when it performs significantly better, switch 100% traffic over to candidate
