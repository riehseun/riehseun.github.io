# Requirement

How many models are there? Over 100
What kind of models? Feed recommendation, harmful content detection, product search, fraud detection
Are inferences batch and real-time? Both
What should be latency for real-time inference? Under 200ms
How many daily active users are there? 1B
How many inference requests does each user make? depends on use case
Should the system perform model deployment like A/B testing? Yes
Should the system support continuous training? Yes
Should the system support online and offline feature store? Yes
Should the system support model registry? Yes
Should the system support monitoring of models and infra? Yes

# Design

Data ingestion -> Data processing -> Raw storage -> Curated storage

Data ingestion
  Kafka can be used for both real-time and batch ingestion
  Real-time ingestion can be micro batching (1-60s) to increase throughput
  Batch ingestion can be of API dump, change data capture of DB

Data processing
  Spark structured streaming

Raw storage
  Append only and immutable

Curated storage
  Store data in parquet format for fast columar access 

Model training -> Model registry -> A/B testing -> Deployment

Model training
  Data parallelism
  Model parallelism
  Pipeline parallelism

Model registry
  Model versions
  Parameters

A/B testing
  50% traffic goes to existing model
  50% traffic goes to candidate model
  Only when candidate model perform significantly better, direct 100% traffic to candidate model
  100% traffic to existing and 100% traffic to candidate is safer but double number of prediction is too costly

Deployment
  Deploy models into different regions such as US East, US West, etc
  This avoids cross-regional latency and single point of failure

User -> Global load balancer -> Regional edge -> API gatwway -> Cache -> Online feature store -> Inference -> Output -> Offline feature store -> Data Lake -> Training

API gateway
  Route to differet models based on request

Cache
  Serve frequent prediction (For example, use Redis)
  
Online feature store
  Store dynamic features for inference
  Stores data into raw storage (before processing) and curated storage (after processing)

Inference
  Dynamic batching (For example, micro-batching per 5-10ms) to increase throughput

Offline feature store
  Stores batch features for continuous training

Data lake
  Stores inference and ground truth to generate metrics to monitor model performance
