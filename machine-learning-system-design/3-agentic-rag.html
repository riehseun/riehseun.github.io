# Requirement

Functional
  What should Agentic RAG do? Generate response upon user query
  What type of query does user input? General query
  Should Agentic RAG generate response based on domain specific knowledge? Yes
  What domain specific knowledge should Agentic RAG contain? Could be company internal information
  Does user query include text, image, video? Text only
  Should response include reference to knowledge (or document)? Yes
  Should chatbox consider previous prompts in the same session? Yes

Non-functional
  Is training data available? Yes
  How fast should the generation be? 1-3s
  Should model be deployed to mobile devices? No
  Should model be continuously trained? Yes
  What is expected QPS? 100
  What is peak QPS? 20-30 average QPS
  How often is knowledge updated? New documents are avaiable every month
  Should the system support multi-turn conversation? Yes
  Should the system personalize response? No

# Design

Plan -> Execute -> Verify -> (if applicable) Terminate

Plan - clasifies query using LLM and generates execution graph
  Simple question -> direct LLM
  Knowledge question -> RAG
  Transactional -> API/DB call
  Complex question -> Decompose input into multi tasks -> Execute each task (RAG) -> Verify -> Terminate (Ex. loop detected, max number of call reached, max cost per query reached, confidence threshold reached)
Execute - run tools
  API calls, DB queries
  Retrieval - select between cheap and strong model based on confidence level of retrieved document, when conflicting documents are retrieved, select best document based on freshness and authoritativeness of documents
  Evaluation - when RAG output degrades measure by offline evaluation, find chunks from poorly generated reponse and mark that chuck as stale
Verify
  Self-critic final generated response
Terminate
  Terminate if cycle is detected in the execution graph or when target confidence is met

# Model

Document -> Chunking -> Embedding -> Vector DB

Chunking
  Fixed-size - fixed number of tokens in each chunk (fixed number of overlapping tokens are allowed)
  Semantic - split text into sentences and create fixed-sized window, then merge windows that have certian cosine similarity
Embedding
  Late chunking - break texts into tokens, compute embeddings for each token, then group tokens into chucks (need long context model)
  Late interaction - compare similarity between query tokens and document tokens one by one, then find highest score (compute intensive)
  SentenceBERT - compute averge embedding of all words in each sentence, feed two sentences into two BERTs at a time, then compute cosine simliarity

Query -> API gateway -> Chat service -> Query understanding -> Retriever -> Re-ranker -> LLM (with system prompt) -> Filter ->

Query understanding
  Correct spelling mistakes
  Additional keywords
  Transform input query into embedding
  Cache - store and read query embedding from cache

Retriever
  Search vector DB and find top k chunks that has closest embedding distance to input embedding using ANN (EX. HNSW)
  HNSW - build graph of chunk embeddings, compare closeness between query and random node versus query and all neighboring nodes on random node, then pick node with highest similarity
  Cache - store and read top k result from cache

Re-ranker
  Rank chunks from retriever
  Cross-encoder - both query and chunk are fed to network, then value bewteen 0 and 1 are produced
      
# Evaluation

Retriever - recall@k, precision@k
Re-ranking - NDCG
LLM - faithfulness, answer relevance, context precision/recall
Overall - summarization score, task success rate

# Discussion

Confidence
  Retrieval - cosine similarity between input query and chunk embedding, normalized per query
  If distribution between top-1 and top-k is flat, we have low-confidence
  If retrieved docuemnts have conflicting information, we have low-confidence
Low-confidence
  Increase k, use different retrieval strategy, re-run ANN
  Then, use stronger model

Supporting images or videos are input
  Encode images (or use a model to compute text summary of image) and store in vector DB, retriever searches for both relevant chunks and images, then final chunk and image are sent to multi-modal LLM

Handle very long query
  Text clean up + text summarization service
