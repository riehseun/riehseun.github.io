Model Serving


# Requirement

How many models are there? Over 100
What kind of models? Feed recommendation, harmful content detection, product search, fraud detection
Are inferences batch and real-time? Both
What should be latency for real-time inference? Under 200ms
How many daily active users are there? 1B
How many inference requests does each user make? 100 on average per day (2M QPS)
What is peak QPS? 20-30x of average QPS
Should the system perform model deployment like A/B testing? Yes
Should the system support continuous training? Yes
Should the system support online and offline feature store? Yes
Should the system support model registry? Yes
Should the system support monitoring of models and infra? Yes

# Design

User -> Global load balancer -> API gatwway -> Model router -> Cache -> Online feature store -> Serving layer -> Output -> Data lake -> Training -> A/B Testing 

API gateway
  Authorization
  Rate limiting

Model reuter
  Route to differet models based on request (or endpoint)
  For recommendation and search ranking models, use candidate generation -> ranking -> re-ranking in serving layer
  For real-time detection models  

Cache
  Serve frequent prediction (For example, use Redis)
  TTL to invalidate stale information
  
Online feature store
  Use key-value store like Redis
  Store user and item features
  Feature versioning to ensure point-in-time correctness

Serving layer
  Load models into memory
  Service predictions via RPC
  Use Tensorflow Serving or Triton Inference Server

Data lake
  Stores inference and ground truth to generate metrics to monitor model performance 

Data pipeline
  Ingestion - Kafka for both real-time and batch
  Processing - Spark Structured Streaming
  Raw storage - append only and immutable
  Curated storage - store data in parquet format for fast columar access

Trainign pipeline
  Offline feature store - Stores transformed features to be re-used
  Model training - data/model/pipeline parallelism
  Model registry - model versions, parameters

A/B testing
  Gradual rollout to candidate model (5% -> 10% -> 25% and so on)
  Only when candidate model perform significantly better, direct 100% traffic to candidate model
  100% traffic to existing and 100% traffic to candidate is safer but double number of prediction is too costly

# Discussion

Availability
  Run multiple replicas of each service in different geo location (active-active)
  Trade off between cost and availability

Reliability
  Store replicas of data in data center in different geo location

Scalability
  Compute - horizontal scaling (based on CPU/GPU usage or traffic patterns)
  Data - paritioning (Ex. consistent hashing)

Performance
  Remove unnecessary features, thus size of feature vector
  Pruning - remove non-contributing connections between neurons or neruons themselves
  Quantization - use less number of bits to store weight values
  Knowledge distillation (only if teacher model is available)
  Use dynamic batching (5-10ms) to increase throughput (trade-off bewteen latency and throughput)
  GPU for large DL models and CPU for lightweight models (trade-off bewteen performance from GPU and cost)

Latency
  Network - 20ms
  Feature fecth - 40ms
  Inference - 30ms
  Post-processing - 10ms
  Circuit breaker when latency increases due to failures on feature store or inference
    Cached predictions
    Simpler backup models

Monitoring
  Model metrics - score distribution, feature drift, data drift
  System metrics - QPS, latency
