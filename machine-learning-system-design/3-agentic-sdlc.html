# Requirement

Functional  
  Should the system include human gates? Yes

Non-functional
  How long can the agent take to generate working app and infra? Few or several hours
  How much token budget is allowed for executing each user prompt? Should be configurable
 
# Design

Product -> Planner -> Develop -> Review -> Test -> Release -> Monitor

Product
  Takes user prompt and create user stories includeing acceptance criteria
  Human approves created user stories, or reject quoting reasons and planner improves them and ask for re-approval

Planner
  Design APIs, test case, DB schema, auth mechanisms, networking
  Human approves proposed design

Develop
  Write small PRs (including unit tests)
  Improves PRs based on feedback from Review agent

Review (developer)
  Reviews PR from developer agents (lint, test coverage, etc)
  Human provides the final approval before merge

Release
  Creates infrasturcture files (Docker, Terraform, etc)
  Improves PRs based on feedback from Review agent

Review (devops)
  Reviews PR from release agents
  Human provides the final approval before merge

Test
  Performs functional tests on APIs (load testing in upper environments)

Release-Review-Test repeats in high environment

Monitor
  Conducts integration with monitoring tools to monitor app and infra
  Executes roll-back upon finding bugs running in production
  Then, agent can invoke bug fixing steps to fix the code, test, and release

# Discussion

Compancy specific codebase 
  Vectorize them and storm them in Vector DB
  Develop and Release agents will be provided with working examples from this DB
  Agent can watch github codebase and insert updated codes into this DB to keep syntax of company-specific codes up to date

Change record creation
  Agent can use information from planning state to generate contents to put into change record
  Use API to create record and move it to different stages
  Agent can contact change record approver and update the content based on feedbacks received

Agent workflow
  All agents can run inside a container of Kubernetes cluster
  Agent orchestrator is continuously running, creates executiong graph as DAG, and create each agent as a Job
  Each agent execute a single task from each skill stored in Github repo
  Upon finishing exeuction, agent stores the execution results in both short term (Ex. Redis) and permanent storage (Ex. S3)
  When all agents successfully complete their tasks, orchestrator shuts down
  Secrets used to interact with LLM, APIs, can be created in K8s Secret where each agent can access secret in its own namespace

Model routing
  Each task is routed to different models based its type and complexity
