Fraud Detection


# Requirement

Functional
  What kind of fraud is this? Fraudulent financial transactions 
  How does fraud occur? Use stolen debit or credit card to make purchases
  Where does fraud occur? In website or app where purchases are made
  Should model explain why transaction is deemed fraud? Yes
  What should system do when detecting fraud? Reject transaction
  What is impact of false positive? Loss of revenue from not making transaction
  What is impact of false negative? Unhappy customer lost money from fraud

Non-functional
  Do we have training data available? Yes, 1M fraud transactions and 100M non-fraud transactions
  Should inference be immediate? Yes
  Should model be continuously trained? Yes
  Should model be deployed to mobile devices? No

# Problem

ML
  Input - person and transaction
  Output - whether transaction is fraud or not
  Binary classification

Business problem
  Increase long-term user engagement, thus increase in revenue

# Design

Transaction -> Reject based on simple rule -> Inference -> Reject or human review or pass

# Data

We have labeled dataset
Also, use result from human review

No need to scale or nomalize if using Ensemble model
Compute "averge metrics" like averge account balance for the past 12 months, as these values are less impacted by distribution drift
  
# Model

Baseline
  Logistic regression to determine if transaction is fraud or not

Architecture
  NN models are not less explainable than tree models
  Use XGBoost - no standardization needed but is expensive to re-train

Training
  Data split
  Split labeled data by time - 80% training set, 20% test set
  Further split 80% training set into - 70% training set, 30% dev set
  Run 5 bins cross-validation
  Make sure each customer Id exist in exactly one of the bins and validation set, to avoid data leakage
  Address class imbalance via undersample/oversample or focus loss

Hyperparemeters
  booster - type of booster to use
  tree_method - method for constructing trees
  n_estimators - number of trees to ensemble
  learning_rate - step size in gradient descent
  reg_lambda - L2 regularization
  reg_alpha - L1 regularization
  max_depth - maximum depth of nodes in each tree
  scale_pos_weight - weight assigned to positive instances

# Evaluation

Offline
  Recall - catching actual fraud is very important 
  Precision - accuracy of prediction is important not to reject valid transaction to annoy users
  PR-AUC - better than ROC-AUC due to class imbalance

Online
  Number of user reports on fraud

# Discussion

Streaming features
  Ingestion - Kafka 
  Processing - Spark Structured Streaming

Feature selection
  Too many features can lead to overfitting, technical debt
  Run XGBoost with a single feature to compute predictive power for that feature
  Rank all features by thier predictive power
  Add first feature to feature set
  Add next feature if the dependecy with the first feature is lower than certain threshold
  Repeat until a pre-defined number of features are collected

Reject based on simple rule
  Blacklisted card
  Country mismatch
  Large amount of transaction

Explainability
  SHAP value

Model monitoring
  Feature PSI - PSI calculated for each feature
  Score PSI
  Kendall's Tau Rank Correlation - measure the changes in the ranking of all features
  ShapBand - measure tha deviation of SHAP values from baseline to inference data for each feature
