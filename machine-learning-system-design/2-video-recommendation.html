Video Recommendation


# Requirement

Should recommendations appear on webpage or mobile devices? Yes
How many videos should appear in recommendation? Unlimited
Should the system support multiple languages? Yes
Does video consist of text description, thumbnail image, tags, comments, likes, shares? Yes
Should the system moderate videos? Yes
Should the system focus on user engagement or content quality? Content quality
Should the system focus on exploration or exploitation? Balance of both
Should recommendation be personalized for each user? Yes

How fast should recommendations be generated? Under 1s
How many videos are there? 1B
How many users are there? 100M
How many times does each user visit the system per day? 5, QPS = (5 * 100M) / (24 * 3600)
Can max QPS be 20-30 times the average? Yes 
Should model be continuously trained? Yes

# Problem

ML 
  Input - a user
  Output - a list of videos ranked based on relevance to user
  Supervised ranking problem

Business
  Maximize user engagement while maintaining minimum target precision (we don't want to just recommend click-bait to maximize user clicks)

# Serving

User -> Cache -> Candidate generation -> Ranking -> Re-ranking -> Output & feedback

Cache 
  Return cached recommendation if latency exceeds 1s
  If not, compute generation and store result in cache (many features are dynamic, thus cached recommendation is likely stale)

Candidate generation
  Invoke ANN between user embedding and video embeddings (stored in vector storage) to come up with 1k candidates
  Find 100 popular and trending videos based on speed of increasing likes, watch times, etc
  Find 10 videos that user had very positive previous engagements
  Combine all these to generate candidates
  Latency ~ 200ms  

Ranking
  Use DNN with feature-cross or transformer based ranking
  Latency ~ 500ms

Re-ranking
  Use cross-encoder
  Latency ~ 300ms

# Data

Positive - if user likes video, leaves positive comments, shares videos, watches at least half of video
Negative - if user dislikes video, don't click recommended video, leaves negative comment

# Feature

Video
  Image - CLIP's visual encoder
  Video - split into frames and user CLIP
  Text description, tags - dislimBERT for multi-language
  Number of likes, shared - normalize
  Comments - sample them, and compute weighted average embedding where recent comments have higher weights and old comments have lower weights

User 
  Device, time of day, month of year - bucketized and one-hot
  Language, country, city - embedding

Temporal
  Previous videos that user engaged (watch, like, etc) - sample and compute weighted average embedding

# Model

Candidate generation - two-tower NN
  User tower encodes all user features into a single embedding
  Video tower encodes all video features into a single embedding
  Dot product betwee two embeddings are 1 if user engaged with video, 0 otherwise
  User engagement can be determined by computing weighted sum of all user engagements, positive or negative
  Use BCE loss

Ranking - transformer-based ranking
  Point-wise learn to rank where each user-video pair gets a score
  
Re-ranking - cross-encoder
  Learn user and video feature interactions using transformer encoder 
  Apply penality to score by taking in diversity, freshness into consideration
    Diversity - penalize if similar videos (close video embedding distancecs)
    Freshness - penalize if old video
  Inference is feasible when number of videos are less than hundreds

# Discussion

Availability
  Replicate each service in different geo regions (trade-off with cost)
Reliability
  Partitiion data into shards, create replicas for each shard, and spread them across different geo regions
Scalability
  Horizontally scale each service based on traffic forecast
Performance
  Model quantization - use fewer number of bits to represent floating number
  Pruning - apply regularization like drop-out to shut down some neurons to make network less dense
Monitoring
  System - QPS, latency
  Model - input data drift, feature drift, score drift

Real-time data ingestion
  Dynamic data (or feature) can be ingested via Kafka
  Feature can be computated via Spark Structure Streaming inside Kafka Topic
  Both raw and curated data are persisted in raw (data lake) and curated storage (offline feature store)
  Curated data is also stored in online feature store
Offline feature store
  Store features for training
  Ex. S3
Online feature store
  Store features for inference
  Use fixed-size window to aggregate features, to ensure point-in-time correctness
  Ex. Redis

Content moderation
  Use a separate classification model to detect harmful contents when users upload videos
  Continue using the model after the video is uploaded since features that contributes to prediction may change after video got uploaded

Exploitation vs exploration
  Use bandits to direct 10-15% of recommendation to exploration (Ex. thompson sampling)

Continuous training
  A/B testing to direct 10-15% of traffic to candidate
  Slowly increase traffic to candidate if it performs better than production  

Cold-start
  New user - recommend popular videos for user's geo location and language, content-based recommendation based on answers from onboarding questions
