Index Exchange

Design real-time AD bidding system like Index Exchange


# Requirement
Sellers send bid request
Exchanges asks buyers to submit bids
Buyers submit bids
Auction takes places
Winning buyer (AD) is returned to seller

What is latency budget for entire auction? 150ms
What is inference latency for models? <1ms
How many buyers are there? 100
How many requests come from sellers per second? 1-2M per seconds
What is peak QPS? 5-10x the average 
How does Exchange make money? winning bid and number of impressions

# Problem

Maximize expected revenue under latency constraint
Input: bid request from seller
Output: winning bid

# High-level

Seller -> Bid Request -> Global Load Balancer -> Regional Edge -> Auction Service (Buyer Selection, Buyer Fanout, Auction) -> Bid Response -> Seller

Bid Request
- ad size, ad format, site URL, geolocation
- seller's timeout

Global Load Balancer
- Route traffic to nearest edge
- Perform regional fail-over
- Can be implemented by Anycast
  - All edge advertise same IP address
  - Policies determine most reachable edge

Regional Edge
- TLS termination: removes handshake, decryption work from downstream servers
- Rate limiting
- DDoS protection

Buyer Selection (2-3ms)
- Trim 100+ buyers down to 5-10
- Candidate generation -> Ranking

Buyer Fanout (2-5ms)
- Simultaneously send requests to buyers
- epoll notices event loop when sockets arrive
- Event loop resume coroutine

Auction (<1ms)
- If highest bid < floor price, return No Winner result to seller
- Buyer with highest bid becomes the winner

# Discussion

Buyer Selection
- Candidate generation
  - Objective: trims 100+ down to 20
  - Data: positive if buyers responded before timeout and bid > floor, negative otherwise
  - Feature
    - Buyer: response rate, time-out rate, bid price, impression rate
    - Seller: ad size, ad format, seller id, geolocation
    - Auction: time of day, day of month
  - Model: LR
  - Loss: BCE 
  - Metric: recall@k (k=20)
- Ranking
  - Objecgtive: trims 20 down to 5
  - Data: bid request and buyers who responded, ranked by their bid value (buyers who don't respond get the last position in the ranking)
  - Feature: same feature set as candidate generation
  - Model: XGBoost
  - Loss: pairwise learn-to-rank (LambdaMART)
  - metric: NDCG@k (k=5)
- Selection bias towards buyers with no historical interaction
  - Exploitation: buyers with historically strong bid and response rate
  - Exploration: new buyers
  - Sampling to include new buyers to candidate generation
    - For example, pick top 15 first using candidate generation
    - Then, random sample 5 out from remaining candidates
      - Create bid distribution for each buyer and sample from distribution
      - For buyers with no historical data, initialize prior distribution using average bid data
- Budget pacing
  - For each buyer, keep track of daily spend and budget
  - If buyer is used all the budget, remove it from candidate generation
  - Also, compute speed at which buyer is spending the budget  
  - If buyer is spending too fast, reduce participation rate to candidate generation

Buyer Fanout
- Async IO: one machine handles tens of thousands of auctions using event loop (Instead of one thread per auction)
  - Socket becomes ready
  - Kernal notified event loop
  - Coroutine resumes
  - Response gets processed
- Drop buyers when cost of waiting exceeds revenue from waiting
  - Expected revenue from waiting
    - Keep buyers overall latency and bid distribution
    - Pre-compute expected value from waiting 1ms, 2ms, 3ms, so on
  - Expected cost of waiting
    - Keep sellers latency distribution
    - Compute probability of missing seller timeout, multipled by highest bid so far
  - Terminate when Expected revenue from waiting < Expected cost of waiting

Availabiity
- If multiple data centers, replicate servers in different geo locations
- Active-Active (multiple regions serving traffic together)

Reliability
- Kafka down: retry when it comes back
- Redis down: backup latest feature values to local in-memory cache
  - When Flink updates Redis, update local cache too

Scalability
- Build for peak load
- Add more servers behind load balancer (horizontal scaling)
- Stateless servers

Latency 
- Cache (No DB calls): buyers id, endpoint

Monitoring
- Latency/QPS
- CPU/memory usage
- Buyer's response rate, time-out rate
- Tools
  - Application: Prometheus
  - Dashboard: Grafana
  - Alerting: PagerDuty

Single region concurrency
- Atomic database update

Multi region concurrency
- Patition data such that each data belongs to one region
- Write is done on only one region and updates are asynchronously replicated to other regions

Real-time data processing
- Ex. impression event, buyers respond with bid, sellers send bid requests
- Event is sent to Kafka
- Flink/Spark to process event data
  - Validate against missing fields, incorrect timestamps
  - Feature computation and aggregation (compute response rate over 1 minute window, average bid in last 5/10/20 minutes)
- Data is persisted in Data Lake (offline store for model training, in parquet) and Redis (online store for model serving, feature vector in redis hash)
- Online feature store holds latest values only
- Need mapping between model version and feature version
  - Model registry: model version, training data, feature used (including versions), model metrics
  - Feature registry: feature version, transformation logic, freshness SLA, owner

Exactly once
- Not needed for request/response because of large size (small duplicate events won't change features too much)
- Needed for billing
  - Use UUID generator
  - Use UUID for each billing event id and use it as primary key

A/B testing (candidate generation, ranking, only A/B test one of them at a time)
- Randomly split traffic between control and treatment 50/50
- Track revenue per bid request, buyer response rate, recall on winning buyer
- Start by 1/99, then 10/90, then 20/80
