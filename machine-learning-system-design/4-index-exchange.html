Index Exchange

Design real-time AD bidding system like Index Exchange


# Requirement
Is the goal to find buyer, who is willing to pay most, for the sellers? Yes
Do sellers make AD request? Yes
Does Index Exchange perform auction? Yes
Does Index Exchange make commission when an auction is won? Yes
Does commission also depend on number of click on the AD? Yes

How many requests come from sellers per second? 10M per seconds (which can be 100x when talking to buyers)
How many buyers are there? 100+
What is latency budget for entire auction? 5-10ms
How fast do buyers return their bid? Varies
What is availablity required? 99.9999%

# Problem

Maximize expected value from each seller and buyer match. E = bid value + number of impressions
Input - request from sellers
Output - winning buyer

# High-level

Seller -> Bid request -> Exchange -> Buyer selection -> Auction request -> Bid response -> Auction -> Winning AD -> Seller

Bid request - 10ms (network)
Buyer selection - 1-2ms (ML)
Auction request - 1ms (network)
Bid response - 1ms (network)
Auction - <1ms (ML/rule)
Winning AD - 10ms (network)

# Discussion

Scalability
- Stateless servers
- No DB calls (everything in cache)

Availabiity
- If multiple data centers, replicate servers in different geo locations
- Edge decides which region to route requests to

Latency 
- Use cached decision when latency budget runs out

Edge
- All requests are handled by edge server close to seller
- Buyers close to edge are preferred
- buyer selection model is deployed to edge server

Cache 
- buyers id, endpoint
- response rate, average bid, timeout rate, win rate

Real-time data processing
- Impression/click event
- Event is sent to Kafka
- Flink/Spark to process event data
  - Validate against missing fields, incorrect timestamps
  - Feature computation and aggregation (compute response rate over 1 minute window, average bid in last 5/10/20 minutes)
- Data is persisted in Data Lake (in parquet format)

Buyer selection
- Feature
  - Seller: site, ad size, ad format
  - Buyer: geoloation, historical response rate, historical win rate, historical time-out rate, historical bid price
  - Auction: 
- Model: ranking problem to select top 5 likely buyers
- Objective: maximize P(response)E(bid)P(no-timeout)
- Selection bias towards buyers with no historical interaction: Multi-arm bandit
  - Exploitation: buyers with historically strong bid and response rate
  - Exploration: new buyers

Auction request
- Fanout to 100+ selected buyers
- Waits for responses
- Slower responses outside of latency budget are dropped
  - Create thread for each buyer: 5 buyers * 10M auctions per second = ? (memory, context switching overhead)
  - Async IO: 1 thread manages thousands of auctions at the same time
