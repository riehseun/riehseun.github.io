Index Exchange

Design real-time AD bidding system like Index Exchange


# Requirement
Is the goal to find buyer, who is willing to pay most, for the sellers? Yes
Do sellers make AD request? Yes
Does Index Exchange perform auction? Yes
Does Index Exchange make commission when an auction is won? Yes
Does commission also depend on number of click on the AD? No, it depends on number of impressions

How many requests come from sellers per second? 10M per seconds
How many buyers are there? 100+
What is latency budget for entire auction? 5-10ms
What is availablity required? 99.9999%

# High-level

Seller -> Bid Request -> Global Load Balancer -> Regional Edge -> Exchange Service (Buyer Selection, Buyer Fanout, Auction) -> Bid Response -> Seller

Bid Request - 10-30ms (network)
Exchange - 1ms
Buyer Selection - 1-2ms (ML)
Buyer Request/Response - 2-5ms (network with timeout)
Auction - <1ms
Bid Response - 10-30ms (network)

# Discussion

Availabiity
- If multiple data centers, replicate servers in different geo locations
- Active-active (both replicas are serving traffic)

Reliability
- Buyer does not respond: temporarily remove from candidate generation until it comes back
- Kafka down: retry when it comes back
- Redis down: backup latest feature values to local in-memory cache

Scalability
- Stateless servers
- Add more servers behind load balancer (horizontal scaling)

Latency 
- Use cached decision when latency budget runs out
- Edge
  - All requests are handled by edge server close to seller
  - Buyers close to edge are preferred 
- Cache (No DB calls)
  - buyers id, endpoint

Monitoring
- Latency
- QPS
- CPU/memory usage
- Buyer's average response, time-out rate

Bid Request
- ad slot, site URL, geolocation
- seller's timeout

Global Load Balancer
- Route traffic to nearest edge
- Perform regional fail-over

Regional Edge
- TLS termination: removes handshake, decryption work from downstream servers
- Rate limiting
- DDoS protection

Auction Service
- Read feature vectors from Redis
- Write events to Kafka

Buyer Selection
- Fanout to all 100+ buyers and waits for responses? Too expensive
- Ranking problem to select top 5-10 likely buyers
- Candidate generation
  - Feature: response rate, time-out rate, seller id, geolocation
  - Model: LR with focus on recall
  - Objective: binary cross entroy (1 if buyer responds within timeout window, 0 o/w)
  - Trims 100+ down to 20
- Ranking
  - Feature: win rate, bid price, impression rate, 
  - Model: XGBoost with focus on precision
  - Objective: maximize E(revenue)
  - Trims 20 down to 5
- Selection bias towards buyers with no historical interaction
  - Exploitation: buyers with historically strong bid and response rate
  - Exploration: new buyers
  - Sampling to include new buyers to candidate generation
    - For example, pick top 15 and sample 5 out from remaining candidates
    - Create reward distribution for known buyers
    - Treat new buyers with uniform distribution
    - Sample 5 from above two sets of distributions
  - At ranking stage, new buyers will have some feature values missing
    - If XGBoost is used, missing values are natively handled
    - If other model is used, missing values can be filled using global statistics

Buyer Fanout
- Async IO: one machine handles tens of thousands of auctions (Instead of one thread per auction)
  - Synchronous call will take lots of time. For example, 
  - def auction(): 
        bid1 = call(A) # 20ms
        bid2 = call(B) # 20ms
        bid3 = call(C) # 20ms
        max(bid1, bid2, bid3) # 60ms for each auction
  - One thread for each buyer will spwan too many threads. For example,
  - def auction():
        thread(A) # 20ms
        thread(B) # 20ms
        thread(C) # 20ms
        wait_for_all() # 20ms but 3 thread for each auction
  - Async IO lets a machine to act on network events. For example,
  - async def auction():
        result = await asyncio.gather(
            call(A)
            call(B)
            call(C)
        )
    asyncio.run(auction)
- Drop buyers when cost of waiting exceeds revenue from waiting
  - Expected revenue from waiting
    - Maintain buyer's response latency distribution, bid distribution
    - Compute expected revenue at certain timeout values by P(response)E(bid)
  - Expected cost of waiting
    - Compute probability of missing seller's timeout
    - P(timeout)E(revenue)

Auction
- Buyer with highest bid becomes the winner

Real-time data processing
- Ex. impression event, buyers respond with bid, sellers send bid requests
- Event is sent to Kafka
- Flink/Spark to process event data
  - Validate against missing fields, incorrect timestamps
  - Feature computation and aggregation (compute response rate over 1 minute window, average bid in last 5/10/20 minutes)
- Data is persisted in Data Lake (offline store for model training, in parquet) and Redis (online store for model serving, feature vector in redis hash)
- Online feature store holds latest values only

Exactly once
- Not needed for request/response because of large size (small duplicate events won't change features too much)
- Needed for billing  
  - Used UUID generator
  - Store UUID for each billing event entry

MLOps
- Model registry: model version, training data, feature used, model metrics
- Feature registry: feature version, transformation logic, freshness SLA, owner
- A/B testing (candidate generation, ranking, only A/B test one of them at a time)
  - Randomly split traffic between control and treatment 50/50
  - Track revenue per bid request, buyer response rate, recall on winning buyer
  - 100/100 duplication is biased because buyers will never see actions from treatment and adopt to it
  - Start by 1/99, then 10/90, then 20/80
