Search Ranking


# Requirement

What do users search for? products that users want to purchase
Should search be personalized for each user? Yes
How many languages should the system support? Multiple languages
Does product consist of text description, image, video, review, comments, likes? Yes
Can users put products into cart without purchasing it? Yes
Should the system focus on user engagement or product quality? product quality
Should the system focus on exploration or exploitation? balance of both
Should the system moderate products? Yes

How fast should search result be? Under 1s
How many products are there? 1B
How many users are there? 100M
How many search does each user execute per day? 10 per day (100M * 10) / (24 * 3600) QPS
Can the peak QPS be 20-30 times the average? Yes
Should model be continusouly trained? Yes

# Problem

ML - given search term and user as inputs, output a ranked list of products (supervised ranking problem)
Business - maximize user engagement subject to certain precision (we don't want to recommend products that are just click baits)

# High-level

User and search term -> Query expansion -> Cache -> Candidate generation -> Ranking -> Re-ranking -> Output -> Feedback -> Re-training

Query expansion
  Trim search terms
  Add additional context around search terms
  For example - spell correction, query embedding, query intent
  Search term embeddings can be computed and stored in inverted index, and retrieved during inference time (Ex. Elasticsearch)
Cache
  If generation takes longer than 1s, return cached search result
Candidate generation
  Trim 1B products into 10M
  Use ANN between user embedding and embedding computed from both search term and product (Ex. HNSW)
Ranking
  Rank search result 
Re-ranking
  Apply additional criteria such as diversity and freshness of products

# Data

Explicit - like/dislike, positive/negative review, report, hide, cart, purchase
Implicit - takes longer than X seconds to view the product
Take a weighted sum of both explicit and implicit feedbacks to compute engagement score

# Feature

User
  Previous products user carted/purchased - take average embeddings of products, discounted by time
  Country, city, language - embedding
Product
  Image, video - CLIP
  Text description - DistlimBERT
  Review - sample and compute average embedding using DistlimBERT
Content
  Time of day, month of year, device - one-hot
Temporal
  Products that user carted/purchased in current session - take average embeddings of products, discounted by time

# Model

Candidate generation - two-tower
  User tower computes user embedding
  Item tower computes embeddings based on product features and search term embedding
  Dot product is taken to find closeness between user and item tower
Ranking - list-wise learn to rank
  Point-wise and pair-wise is less compute-intensive, but list-wise has better accuracy
Re-ranking
  Freshness - penalize old products 
  Diversity - penalize products with similar embeddings

# Evaluation

Candidate generation
  Recall @ k
Ranking
  NDCG

Click-through rate
Increase in total time that user spends in platform
Increase in total number of products carted or purchased

# Discussion

Availabilty
  Run multiple replcas of each service in different geo location (trade-off with cost)
Reliability
  Partion data in shards and store multiple replca for each shard in different geo location
Scaliability
  Horizontally scale each service based on traffic pattern
Performance
  Cached search result
  Model optimization - quantization (fewer bits to represent number), pruning, distillation (if teach model is available)
Monitoring
  System - latency
  Model - feature drift, input data drift

Offline feature store
  Process user feedbacks (interaction data) as features and store them in offline feature store, to be used for re-training
Online feature store
  Dynamic user and item features are read from online feature store during inference time
  Transformation logic for offline and online store must be the same, not to degrade the model over time
