# Requirement and Problem

Functional
  What do users search? Products in commercial webiste?
  What does a product consist of? Image, video, description, price, category, inventory, tag, review, comment
  Should the system consider next delivery date for out-of-inventory items? Yes
  Should the system multiple language? No, English only
  How many products should each search show? Up to 100
  Should the system moderate product image, video, description? Yes
  Should the system moderate language used in comments? Yes
  Should the system prioritize exploration and exploitation? Up to you
  Should the system prioritize user engagement and purchase or quality of products? Up to you
  Should search result be personalized to each user? Yes

Non-functional
  How fast should search result be? Under 200ms
  How many users are there? 100M daily users
  How many total products are there? 10B
  How often are products being updated? Once in 6 months on average
  Should model be continously trained? Yes

ML problem
  Input - query
  Output - list of products ranked by relevance to query
  Supervised ranking problem

Business problem
  Maximize total amount of user purchase, subject to certain precision (system shouldn't just recommend bait products to maximize short-term revenue)

# High-Level Design

Query -> Query understanding -> Embedding service -> Inverted index
Product -> Text/Image/Video encoder -> Product Embedding
Query -> Cache -> BM25 -> Candidate generation -> Ranking -> Re-ranking -> Search result -> Data Lake -> Re-training

# Data and Feature Engineering

Training data
  Positive - if user save product for later, stores product into cart, purchase product
  Negative - if user clicks the product but performs no further action or impression only
  Hard positive - if user purchases product and later leave good review or comments
  Hard negative - if user purchases product and later leave bad review or comments

Features
  User - Products that user previously purchased (use average product embeddings discounted by time)
  Product - description, image, video, review, comments, tag, price, number of inventory, next delivery date
  User-Product interaction - purchased, carted, saved, impression, comments, review
  Context - device, time of day
  Temporal - previous search query, previous product that user clicked and spent X amount of time or saved or carted or purchased
  Query -
  Query-item cross

Feature engineering
  Text (description) - encoder-only models like BERT 
  Text (tag) - CBOW (semantic understanding not needed)
  Text (comments) - sample comments and compute averge embeddings of sampled comments
  Numerical (price, umber of inventory, review) - standardize
  Context (device, time of day) - bucketize and one-hot
  Temporal - average embeddings

# Model Development

Baseline
  Heuristic - rank products by reviews
  Logistic regression - binary classification on whether user will purcahse product

Architecture
  Candidate generation
    Two-tower network
    User tower computes user embedding
    Item tower computer item embedding
    Take dot product where ground truth of 1 represents whether user saved, carted, purchsed product. 0 o/w
    Create different classification head for save, cart, purchase
  Ranking
    XGBoost
  Re-ranking
    Diversity - penalize products from same vendors
    Freshness - penalize old products

Traning
  Ranking
    Loss - take weighted linear combination of individual losses (likelihhood of save, cart, purchase)

# Detailed Design

Real-time user-product interaction data
  Use Kafka for data ingestion and Spark Structured Streaming for data processing
  Store computed features in online feature store (Ex. Redis) for fast read

Product features
  Compute features and store them in offline feature store for training 

Product update
  Extract features
  Generate new embedding
  Update offline feature store

Cache
  Store search results for each user for each query in cache, with TTL 

Query understanding
  Spell correction
  Query embedding
  Query intent
  Product category

Monitoring
  Store both inference and ground truth in data lake to monitor model perfomance
  Monitor feature distribution drift
