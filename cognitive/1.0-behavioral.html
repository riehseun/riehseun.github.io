Elasticsearch cluster migration (TD Layer6 AI)

  • S
    • Database for storing inference and ground truth data

  • P
    • Elasticsearch was outdated at version 6
    • It was free version and did not support authentication
    • Write was slow, taking more than 3 hours during inference and ground truth
    • Disk was nearly full
  • S
    • Upgraded Elasticsearch version from 6 to 8
    • Capacity estimate for the next 5 years
    • Deployed the cluster in kubernetes because model training and inference were also running in Kubernetes, and the integration could become smoother
    • Higher compute and lower storage to hot nodes and lower compute and higher storage for cold nodes
    • Splited each Elasticsearch index into many shards to achieve parallet reads (Each shard did not exceed 600MB)

    • Took a periodic snapshot of data and store them into storage account to achieve reliability
  • I
    • 2x in read/write
  • L
    • Official documentation may not be fully correct
    • In particular, taking snapshot of data from version 6 and restoring it in version 8 did not work. Thus, I had to leaverage reindexing
    • Learned to leave lots of lead time for implementation to handel potential issues

Migrating to Job Clusters for Azure Databricks (TD Layer6 AI)

  • S
    • All ML use cases were using interactive clusters for inference workload
    • Interactive clusters will shut down only after they are not used for more than 10 minutes, during which time the cost will incur
    • Accumulated costs from all interactive clusters for all ML use cases were significant, around 5k per month
    • I wanted to use job clusters because they would terminate immediately after job finishes
  • P
    • Inference pipeline was executed in Azure DataFactory and the most straight-forward way was to configure job clusters on ADF UI for each use case
    • However, TD internal process demanded that any change to ADF must be maded by TD internal pipeline and go through change approval process
    • Going through TD process for each use case whenever I needed to make changes to cluster configuration was unrealistic
  • S
    • I designed a workaround such that instead of configuring job clusters on ADF UI, I store configuration files for each use case in DBFS
    • When inference is run on ADF, it would first read corresponding configuration file in DBFS, and create and run jobs based on that configuration
    • When I needed to make changes to cluster configurations, instead of deploying ADF changes for all use case, I could just deploy one notebook in ADB which creates the configuration files in DBFS
    • The workaround was accepted by the team and team successfully started using job clusters
  • I
    • 47k - 70k cloud cost savings for each use case in production (9 use cases in total in production)
  • L
    • Sometimes, well thoughtout workaround is necesasry when process makes engineering impossible
