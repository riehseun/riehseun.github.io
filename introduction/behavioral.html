Past learnings

  • What business told me was rarely what they actually wanted. Requirements were never written in a way that delivered context. I realized I needed to interpret the requirements correctly by contiunously asking questions.
  • Mixing up agile and waterfall was a mistake because we had fixed deliverables by certain date while allowing new items to come into the sprint. And we end up with infinite amount of work. If items come in, then some items need to move out.
  • Manual testing and deployment hit the limit quickly as the application became more complex. We insisted "due deligence" of human effort but we quickly realized that everyone makes mistakes. Automation is not only recommended, it is necessary.
  • When we were estimating the work effort, giving a range of time only made business assume the shortest. I learned to consider the worst case scenario to always make promise that I can deliver.

ML infrastructure deployment automation (Layer6 AI)

  • MLOps team was manually constructing 500 yaml files to provision infrastructure for machine learnig use cases. 95% of information we provided were duplicate.
  • I wanted to automate this manul work.
  • I came up with using one configuration file where we supply only 5 percent information. Then I wrote Python scripts to automatically generate all those yaml files based on that 1 configuration file.
  • Use case infrastructure provisioning, which used to take 2 weeks were reduced to less than 1 week.

Enterprise DevOps pipeline feature enhancement (Wealth SDDE)

  • When users made mistakes in the pipeline configurations, they had to raise a new Pull Request. There was no way update deployment configuration for inflight releases. This caused delays between few days to one week in deployments.
  • I wanted to allow users to update pipeline variables without going back to the beginning.
  • I could not allow users to update variables in the tool itself because audit claimed that the source of truth must come from GIT. So, I came up with a way to allow users to invoke a separate release which targeted the original release to update its deployment variables. This design passed the audit because that separate release would be triggered based on GIT, which we treat as source of truth.
  • User feedback was their deployment time got reduced by twice on average because they didn't have to go back to the beginning each time they realize mistakes in the middle of the releases.

  • We used Jenkins as a tool for continuous integration. Job were created manually where scripts were embedded within the jobs. Each job was different, making it very hard to manage the pipeline for each project.
  • I wanted systematic approach to setup the pipeline for all projects.

Made tradeoffs in your projects / Made short-term sacrifice for long term gain

Log4J (Layer6 AI)

  • Log4j issue hit Layer6 AI hard affecting most containers running in production in December 2021. The issue came during the weekend, so we had limited responses from the users. If we kept the containers running, we would have serious security risks of RCE (Remote Code Execution). But if we just shutdown the containers, it could impact businesses.
  • I wanted to remediate log4j situation as soon as possible.
  • I tried to reach out to all users. But seeing people didn't respond, I took an intiative to tell my team that we should shutdown all the containers. It was because I determined the risk of RCE far outweighted the loss of business outputs in the next several days.
  • I took 3 days (including the weekend) to patch all the containers and re-stood them up. Although that caused minor business outage where few use cases that ran daily batch inference and ground truth could not run for few days, I still feel it was the right decision to make since Layer6 AI was not exploited by this vulnerability.
