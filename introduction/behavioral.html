Past learnings

  • What business told me was rarely what they actually wanted. Requirements were never written in a way that delivered context. I realized I needed to interpret the requirements correctly by contiunously asking questions.
  • Mixing up agile and waterfall was a mistake because we had fixed deliverables by certain date while allowing new items to come into the sprint. And we end up with infinite amount of work. If items come in, then some items need to move out.
  • Manual testing and deployment hit the limit quickly as the application became more complex. We insisted "due deligence" of human effort but we quickly realized that everyone makes mistakes. Automation is not only recommended, it is necessary.
  • When we were estimating the work effort, giving a range of time only made business assume the shortest. I learned to consider the worst case scenario to always make promise that I can deliver.

Most challenging experience / Most complex project

Elasticsearch cluster migration (Layer6 AI)
  • We migrated Elasticsearch database from version 6 to 8 to improve the machine learning platform. The old cluster was deployed in 3 virtual machines where read/write speed was slow. Also, the disk was nearly full.
  • I took charge of deploying new Elasticsearch database cluster in Kubernetes and migrated data from old cluster to new cluster.
  • I did capacity estimate in terms of compute and storage for the next 5 years. Then, I designed compute and storage for each type of node in the cluster. For example, frequently used node would have higher compute but lower storage and rarely used node would have lower compute but higher stroage. I deployed statefulsets, services, persistent volumnes, secrets, and custom DNS in Kubernetes.
  • The new cluster was 10 times faster in write operation and 3 times faster in read operations. The new cluster was about twice more expensive when it comes to cloud billing but new cluster had about 5 times more disk.

Caculated risk and take / Made tradeoffs in your projects / Made short-term sacrifice for long term gain

Log4J (Layer6 AI)
  • Log4j issue hit Layer6 AI hard affecting most containers running in production in December 2021. The issue came during the weekend, so we had limited responses from the users. If we kept the containers running, we would have very serious security risks of Remote Code Execution. But if we just shutdown the containers, it could impact businesses.
  • I wanted to remediate log4j situation as soon as possible.
  • I tried to reach out to all users. But seeing people didn't respond, I took an intiative to tell my team that we should shutdown all the containers. It was because I determined the risk of RCE far outweighted the loss of business outputs in the next several days.
  • It took 3 days to patch all the containers and re-stood them up. Although that caused minor business outage where few use cases that ran daily batch inference and ground truth could not run for few days, I still feel it was the right decision to make since Layer6 AI was not exploited by this vulnerability.

Enterprise DevOps pipeline feature enhancement (Wealth SDDE)
  • When users made mistakes in the pipeline configurations, they had to raise new Pull Requests. There was no way update deployment configuration for inflight releases. This caused delays between few days to one week in deployments.
  • I wanted to allow users to update pipeline variables without going back to the beginning.
  • I could not allow users to update variables in the tool itself because audit claimed that the source of truth must come from GIT. So, I came up with a way to allow users to invoke a separate release which targeted the original release to update its deployment variables. This design passed the audit because that separate release would be triggered based on GIT, which we treat as source of truth.
  • User feedback was their deployment time got reduced by twice on average because they didn't have to go back to the beginning each time they realize mistakes in the middle of the releases.
