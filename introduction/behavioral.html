Past learnings

  • What stakeholders told me was rarely what they actually wanted. Requirements were never written in a way that delivered context. I realized I needed to interpret the requirements correctly by contiunously asking questions.
  • Mixing up agile and waterfall was a mistake because we had fixed deliverables by certain date while allowing new items to come into the sprint. And we end up with infinite amount of work. If some items come in, then some items need to move out.
  • Manual testing and deployment hit the limit quickly as the application became more complex. We insisted "due deligence" of human effort but we quickly realized that everyone makes mistakes. Automation is not only recommended, it is necessary.
  • When we were estimating the work effort, giving a range of time only made stakeholders assume the shortest. I learned to consider the worst case scenario to always make promise that I can deliver.

Most challenging experience, Most complex project, Really hard problem

Elasticsearch cluster migration (Layer6 AI)
  • We had a database for storing inference and ground truth data, called Elasticsearch. The version 6 that we were using had serveral problems. We were using open source version, which did not support authentication. The database cluster was deployed in 3 virtual machines and read/write was slow. Also, the disk was nearly full.
  • I had to migrate this Elasticsearch database from version 6 to 8 and also apply the Enterprise license we purchased.
  • First of all, I had to do some capacity estimate to figure out compute and storage for the next 5 years. The way I designed it that nodes that are used will have higher compute but lower storage and nodes that are used less frequently will have lower compute but higher stroage. I decided to deploy the cluster to Kubernetes because the we computed inference and ground truth in Kubernetes. So we have native integration between the services and expect lower latency.
  • The new cluster was 3 times faster in write operation and twice faster in read operations. The new cluster was twice more expensive but it had twice more disks.
Enterprise DevOps pipeline feature enhancement (Wealth SDDE)
  • We were using Enterprise DevOps pipeline for all deployments in the bank. When users made mistakes in the pipeline configurations, they had to go back to the beginning and raise new pull Requests. There was no way update deployment configuration in the middle of releases.
  • I really wanted to allow users to update deployment configurations without going back to the beginning.
  • I could not allow users to update configurations in the tool itself because audit in the bank said that the source of truth must come from GIT. So, I had to design such that users will raise a new pull request and that pull request contains the updated configurations. When this new pull request in merged, a new release get triggeres to update configuration of the existing release using the configuration from the new pull request. And this works because the new pull request has the unique identifer of the existing release. And this design passed the audit.
  • Users started using this feature a lot and general feedback was deployment time got reduced around twice.

Caculated risk and take, Made tradeoffs in your projects, Made short-term sacrifice for long term gain, Had to make an unpopular decision

Log4J (Layer6 AI)
  • Log4j issue hit Layer6 AI hard affecting most containers running in production in December 2021. The issue came during the weekend, so we had limited responses from the users. If we kept the containers running, we would have very serious security risks of Remote Code Execution. But if we just shutdown the containers, it could impact businesses.
  • I wanted to remediate log4j situation as soon as possible.
  • I tried to reach out to all users. But seeing people didn't respond, I took an intiative to tell my team that we should shutdown all the containers. It was because I determined the risk of RCE far outweighted the loss of business outputs in the next several days.
  • It took 3 days to patch all the containers and re-stood them up. Although that caused minor business outage where few use cases that ran daily batch inference and ground truth could not run for few days, I still feel it was the right decision to make since Layer6 AI was not exploited by this vulnerability.
