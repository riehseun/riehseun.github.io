Past learnings

  • What business told me was rarely what they actually wanted. Requirements were never written in a way that delivered context. I realized I needed to interpret the requirements correctly by contiunously asking questions.
  • Mixing up agile and waterfall was a mistake because we had fixed deliverables by certain date while allowing new items to come into the sprint. And we end up with infinite amount of work. If items come in, then some items need to move out.
  • Manual testing and deployment hit the limit quickly as the application became more complex. We insisted "due deligence" of human effort but we quickly realized that everyone makes mistakes. Automation is not only recommended, it is necessary.
  • When we were estimating the work effort, giving a range of time only made business assume the shortest. I learned to consider the worst case scenario to always make promise that I can deliver.

Achievements

Most challenging experience / Most complex project

Elasticsearch cluster migration (Layer6 AI)
  • The team wanted to migrate database solution from major version 6 to 8 in order to leverage the newest functionality.
  • I took charge of deploying new database cluster in Kubernetes and migrate data from old database cluster, which was deployed in a set of virtual machines.
  • I performed analysis to estimate capacity in terms of compute and storage for the next 5 years. Then, I came up with proper compute and storage requirement for each type of node in the cluster. For example, frequently accessed node would have high compute but lower storage and rarely assessed node would have lower compute but higher stroage. Then, I deployed the new cluster in development environment, then tested it by connecting applications to the new database and running business logcis. The deployment was executed using Helm chart, which included statefulset, services, persistent storage, credentials, certificate, and custom DNS.
  • The newly deployed cluster was 10 times faster in write operation.
ML infrastructure deployment automation (Layer6 AI)
  • MLOps team was manually constructing 500 yaml files to provision infrastructure for machine learning use cases. 95% of information we provided were duplicate.
  • I wanted to automate this manul work.
  • I came up with using one configuration file where we supply only 5 percent information. Then I wrote Python scripts to automatically generate all those yaml files based on that 1 configuration file.
  • Use case infrastructure provisioning, which used to take 2 weeks were reduced to less than 1 week.
Enterprise DevOps pipeline feature enhancement (Wealth SDDE)
  • When users made mistakes in the pipeline configurations, they had to raise new Pull Requests. There was no way update deployment configuration for inflight releases. This caused delays between few days to one week in deployments.
  • I wanted to allow users to update pipeline variables without going back to the beginning.
  • I could not allow users to update variables in the tool itself because audit claimed that the source of truth must come from GIT. So, I came up with a way to allow users to invoke a separate release which targeted the original release to update its deployment variables. This design passed the audit because that separate release would be triggered based on GIT, which we treat as source of truth.
  • User feedback was their deployment time got reduced by twice on average because they didn't have to go back to the beginning each time they realize mistakes in the middle of the releases.
EDP local development environment (Wealth SDDE)
  • The development team had manually set up the local development environment. It took 3 months on average for a new developer to fully setup the complex pipeline environment in his/her local machine. Also, there were some differences between each developer's local environment, which caused bugs in the code.
  • I wanted to overcome this problem.
  • I took an initiave to fully automate provisioning and configuration of all pipeline tools. I wrote docker-compose file to deploy containers for tools like BitBucket, Jenkins, XL Release, XL Deploy, and Postgres SQL. I wrote bash scripts to configure all these tools in fully automated way so that everyone in the team would have the exact same local development environment.
  • New users started taking only 2 weeks on average to be on-boarded because they did not have to go through configuring all the details of local development environment.

Build something from scratch

Multibranch pipeline (RESL & HOJ)
  • We used Jenkins as a tool for continuous integration. Jobs were created manually where scripts were embedded within the jobs. Each job was different, making it very hard to manage the pipeline for each project.
  • I wanted systematic approach to setup the pipeline for all projects.
  • I wrote multibranch pipeline from scratch using Groovy script so that every project would use the same codebase for continuous integration. And the codebase is stored and updated in one place rather than each job being managed differently.
  • Each project in the line of businss reduced time to onboard to continuous integration from 1 week to 1 day on average. Because we established a streamlined way to configure the hooks from code repo to Jenkins and how to configure pipeline for each project in a standardized way.

Made current system better

Enhance Kubernetes cluster (Layer6 AI)
  • The Kubernetes clusters we were using for model development and serving were not robust because they lacked monitoring and security.
  • I wanted monitoring and security integrated with K8s clusters.
  • I used helm charts to deploy datadog, kube-state-metric, and Kubernetes Reboot Daemon for monitoring, collecting metrics, and to automatically reboot nodes after the cloud provider applied patches.
  • The time to notice CPU and memory outage was cut by 1 day on average and we were able to solve problems before users could even notice. The time and effort to reboot nodes manually by checking the condition inside the machines have been eliminated.

Had to manage unexpected changes in deadlines

VMC pipeline (Wealth SDDE)
  • I was working on long term development of pipeline to provision and configure infrastructure on VMWare cloud. But, I was suddently given a very tight deadline, which was 2 weeks. The release date was already communicated to the users in the Enterpise by the leadership without consulting the first line staff in the engineering.
  • I wanted to build and release VMC provisioning feature in the pipeline in 2 weeks.
  • I gathered all the requirements from the stakeholders. I realized it was impractical to target all of them. I suggested the team to deliver the requirements in phases. First of all, I would finish the infrastructural provisioning for various middlewares (jboss, springboot, nodejs) including the core configuration of those infrastructure because those were most critical. Then, I would target allowing users to provide their own custom configuration for the infrastructure in the second phase. Then, the ability to cleanup any unused infrastructure would come in third phase.

Made tradeoffs in your projects / Made short-term sacrifice for long term gain

Log4J (Layer6 AI)
  • Log4j issue hit Layer6 AI hard affecting most containers running in production in December 2021. The issue came during the weekend, so we had limited responses from the users. If we kept the containers running, we would have serious security risks of RCE (Remote Code Execution). But if we just shutdown the containers, it could impact businesses.
  • I wanted to remediate log4j situation as soon as possible.
  • I tried to reach out to all users. But seeing people didn't respond, I took an intiative to tell my team that we should shutdown all the containers. It was because I determined the risk of RCE far outweighted the loss of business outputs in the next several days.
  • I took 3 days (including the weekend) to patch all the containers and re-stood them up. Although that caused minor business outage where few use cases that ran daily batch inference and ground truth could not run for few days, I still feel it was the right decision to make since Layer6 AI was not exploited by this vulnerability.
