Requirement

Functional

  • What are the most important features? Upload/download files, sync files, send notification
  • Is this mobile app or web app? Both
  • What are supported file formats? Any file type
  • Do files need to be encrypted? Yes
  • Is there a file size limit? Yes, 10GB
  • How many users? 100M daily users

Non-functional

  • Availability
  • Reliability
  • Scalability
  • Performance
  • Consistency
Reference: System Design Interview, Alex Xu

Estimation

Server

  • Assume
    • 100M active users
    • A single server can handle 10k connections per day
  • 100M / 10k = 10,000 servers are needed

Storage

  • Assume
    • 100M active users
    • Each user uploads 2 files per day
    • Each file is 1MB
  • 2 files per day * 100M * 2 * 1MB = 200TB per day

Bandwidth

  • Assume
    • Read/write ratio is 1:1
  • Incoming - 200TB per day * 8 bits / 86400 seconds per day = 18.51Gbps
  • Outgoing - 18.51Gbps
Reference: System Design Interview, Alex Xu

Storage schema

  • RDBMS is chosen to achieve strong consistency
  • Strong consistency is needed for all users to see the same file information

Relational DB

  • Block

    • user_id (int)
    • last_logged_in_at (datetime)

High level design

User

  • Can either be browser or mobile app

Block servers

  • Upload blocks to cloud storage
  • Each file is split into blocks, that gets a unique hash value
  • The hash values are stored in metadata DB
  • To reconstruct a file, blocked are joined in a particular order
  • Block size is typically 4MB

Cloud storage

  • Amazon S3 is chosen

Cold storage

  • Storage used for inactive data

API servers

  • Authenticates users
  • Updates metadata

Metadata DB

  • Stores metadata of user, file, block, file version

Metadata cache

  • Some metadata is cached for fast retrieval

Notification service

  • Pub-sub like Kafka
  • Notifies clients when files are added/updated/removed so they can pull the latest change

Offline backup queue

  • When clients are offline, information is stored so that it can be synced when clients come back online

API

  • upload_file(data, upload_type)
    • data
      • Local file to be uploaded
    • upload_type
      • simple - when the file size is small
      • resumable - when the file size is large and network interruption is likely
  • download_file(path)
  • get_file_revision(path, limit)
    • limit
      • Max number of revisions to retrieve

Detailed design

API gateway

  • Intercept all client requests
  • For example, edit request, comment on document, authentication, notification, data storing request

App server

  • Document format conversion
  • Import/export document

Operations queue

  • Small data caused by editing document is queued to be processed
    • Characters, images, videos, comments are added to the queue
  • WebSockets is used to observe live changes to the document
    • Using HTTP to send lots of minor changes is inefficient

Session server

  • Maintains document privilege of users

RDBMS

  • Stores user information, document information

NoSQL

  • Stores user comment

Time series DB

  • Stores edit history

Blob storage

  • Stores images and videos in the document

Redis

  • Cache to store
    • User session
    • Typeahead features
    • Frequently accesses documents

CDN

  • Cache images and videos

Evaluation

Availability

  • Replication
    • Replicate each server and DB
    • Replicate entire components to different zone or geographical region
  • Fail over

Reliability

Scalability

  • Horizontal scaling
  • Data replication
    • Horizontal sharding of RDBMS
  • Different DB for different purpose

Performance

  • Redis
  • CDN
  • Websocket instead of HTTP
  • Async operation for email, notification, view count, comment

Consistency

  • Strong consistency - everyone must see the same ordering of characters in the document
  • Time series DB preserves order of events
  • OT or CRDT can be used to resolve conflicts and push result to time series DB
Reference: System Design Interview, Alex Xu

Distinct component

Operational transformation (OT)

  • Commutativity - the order of applied operations does not affect the end result
  • Idempotency - similar operations that have been repeated apply only once
  • If conflict, OT resolves conflict, then pushes corrected state to end users

Conflict-free replicated data type (CRDT)

  • Assigns globally unique identifier to each character
  • Globally orders each character
