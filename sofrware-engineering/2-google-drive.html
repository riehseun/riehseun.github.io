Google docs

Requirement

Functional

  • Multiple users can view/edit documents simultaneously.
  • Chaged made by a user should be propagated to other users.
  • Resolve conflicts when many users are updating the same portion.
  • Users can see view count of the document.
  • Users can see history of collaboration of the document.

Non-functional

  • Availability.
  • Reliability.
  • Scalability.
  • Performance.

Estimation

Server

  • Assume
    • 100 M active users.
    • A single server can handle 10k connections per day.
  • 100M / 10k = 10,000 servers are needed.

Storage

  • Assume
    • Each user creates 1 doc per day.
    • 100M active users.
    • Each doc is 1MB.
  • 1 doc per day * 100M * 1MB = 100TB per day.

Bandwidth

  • Assume
    • Each user views 10 doc per day.
    • 10 * 100M / 86400 = 11574 doc views per second.
  • Incoming: 100TB per day * 8 bits / 86400 seconds per day = 9.26Gbps.
  • Outgoing: 11574 * 1 MB * 8 = 92.6bps.

High level design

  • When users collaborate, request is forwarded to operations queue. Conflict gets resolved here and data is batched.
  • Different versions of document is stored in time series DB.
  • Async operations like view count and comments are queued in pub-sub like Kafka.
  • App server supports document import/export.

Component design

API gateway

  • Intercept all client requests.

Time series DB

  • Stores edit history.

Blob storage

  • Stores images and videos in the document.

Operations queue

  • Small data caused by editing document is queued to be processed.

Session server

  • Maintains document privilege of users.
