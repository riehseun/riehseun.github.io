Pub-sub

  • Pub-sub

Pub-sub

  • Asynchronous service-to-service communication.
  • Producers and consumers are disconnected and operate independently.
  • Enables push-based notification.
  • Handles log ingestion.

Requirement

Functional

  • Create and subscribe to topic.
  • Read, write, and delete message.
  • Specify retention time.

Non-functional

  • Availability
  • Durability
  • Scalability
  • Fault tolerance

High level design

Broker

  • Where producers send messages to and consumers read messages from.

Cluster manager

  • Monitors broker.
  • Broker and topic registry
    • Stores list of topics for each broker.
  • Replication
    • Replication by the leader-follower approach.
    • If the leader fails, the cluster manager decides who the next leader is.
    • If follower fails, it adds a new broker.

Consumer manager

  • Verify the consumer
    • Verify if the consumer is allowed to read a certain message.
  • Retention time management
    • When exceeds the retention time, the message should be inaccessible to the consumer.
  • Message receiving options management
    • System pushes the data to its consumers.
    • Consumers request the system to read data from a specific topic.

API

  • create(topic_id, topic_name)
  • write(topic_id, message)
  • read(topic_id)
  • subscribe(topic_id)
  • unsubscribe(topic_id)
  • delete_topic(topic_id)
