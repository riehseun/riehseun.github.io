Rate limiter

  • Prevent resource starvation. (DDOS attack)
  • Manage policies and quotas.
  • Avoid excess costs.

Throttling

  • Hard: for example, when exceed 500, the request is discarded.
  • Soft: for example, allow 5% beyong the limit, such as 525.
  • Dynamic: for example, allow additional requests as long as there is free resource.

Where to place rate limiter

  • Client: not safe and configuration can be difficult.
  • Server: a viable option.
  • Middleware: between client and server.

Requirement

Functional

  • Limit number of requests a client can send within time window.
  • Make the number of requests within time window configurable.
  • Clients should get error message when threshold is reached.

Non-functional

  • Availability
  • Low latency
  • Scalability

High level design

Component design

Throttle rule DB

  • Stores rules, where each rule has number of requests allowed for each client per unit time.

Throttle rule retriever

  • periodically checks for modifications of the rules in DB.

Throttle rule cache

  • Cache frequently retrived rules from DB.

Client identifier builder

  • Generate unique identifier for each request.
  • This ID is used as key in DB.
